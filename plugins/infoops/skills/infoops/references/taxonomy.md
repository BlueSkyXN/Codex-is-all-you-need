# 数据源分类法

InfoOps 管理的所有信息源的分类规则、排除规则、标签体系和生命周期。

---

## 1. IM 数据源分类

### 群组分类

| 分类 | 定义 | 目录 | ID-MAP 分区 |
|------|------|------|-----------|
| 内部群 | 成员全部为本企业员工的群 | sources/groups/internal/ | Internal Groups |
| 外部群 | 含外部联系人、或挂在外部组织下的群 | sources/groups/external/ | External Groups |

分类依据是群的属性，不是内容。一个内部群讨论外部客户问题，仍然是内部群。

### 内外部群的自动判断

Agent 判断一个群是内部还是外部时，执行以下步骤：

1. `lark-cli im chats get --chat-id <chat_id> --as user` 获取群信息
2. 检查返回的 `chat_type` 和 `tenant_key` 字段
3. 如果无法从群信息判断，拉取成员列表：`lark-cli im chat.members get --chat-id <chat_id> --as user`
4. 检查成员中是否有 `tenant_key` 不同于自己的用户 → 有则为外部群
5. 无法确定时标记为内部群，等人工确认

### 单聊分类

| 分类 | 定义 | 追踪方式 | ID-MAP 分区 |
|------|------|---------|-----------|
| 内部单聊 | 与本企业同事的单聊 | 通过 people/ 人物档案 | Internal Chats |
| 外部单聊 | 与外部联系人的单聊 | 通过 people/ 人物档案 | External Chats |
| Bot 单聊 | 与 Bot 的对话 | 通常不追踪 | Bots |

单聊不在 sources/ 下建独立目录。单聊信息通过 people/ 人物档案管理——人是实体，单聊只是通道。

### 群组档案的内外部差异

| 规范项 | 内部群 | 外部群 |
|--------|--------|--------|
| DIGEST 内容粒度 | 可包含内部决策、内部人名 | 脱敏处理，标注敏感边界 |
| ROSTER 信息深度 | 可写职级、部门、内部角色 | 只写公开身份和群内行为 |
| PROFILE 沟通规则 | 正常 | 必须标注外部群注意事项 |
| 线程引用 | 可直接引用 | 标注 `[外部]` 提醒 Agent 注意措辞 |

### 采集优先级

| 优先级 | 范围 | 判定方式 |
|--------|------|---------|
| 高 | 7天内**我有发言**的群 | `+messages-search --sender <我的open_id> --start <7天前>` 或 `+chat-list --sort active_time` 交叉判定 |
| 中 | 有独立档案但近7天我未发言的群 | sources/groups/ 下有文件 |
| 低 | PULSE 索引中的其他群 | 按需或事件触发 |
| 不采集 | 其余群 | 只在 ID-MAP 中 |

单聊同理：30天内我有发言的人优先。

## 2. 云空间数据源分类

用户术语约定：
- **空间 / drive** = 飞书云空间整体，不叫"云文档"
- **云文档** = 空间下的 docx 类型文件
- **电子表格** = 空间下的 sheet 类型文件
- **多维表格** = 空间下的 bitable 类型文件
- **PDF** = 空间下的 PDF 文件

### 目录映射

| 飞书文件类型 | InfoOps 目录 | 标签 |
|-------------|-------------|------|
| 云文档 (doc/docx) | sources/space/docs/ | `[SD:<名称>]` |
| 电子表格 (sheet) | sources/space/sheets/ | `[SS:<名称>]` |
| 多维表格 (bitable) | sources/space/bitables/ | `[SB:<名称>]` |
| PDF | sources/space/pdfs/ | `[SP:<名称>]` |

### 云知识库 (Wiki) 的处理

Wiki 不作为独立分类。Wiki 中的每篇文章按其实际文件类型归入对应目录：

```
Wiki 文章「<文章名>」 → 类型是云文档 → sources/space/docs/
Wiki 文章「<文章名>」 → 类型是电子表格 → sources/space/sheets/
```

在追踪文件中标注来源：

```markdown
> 类型: Doc (Wiki) | wiki_token: <token> | doc_token: <token>
```

`(Wiki)` 标注表示此文件来自知识库，token 同时记录 wiki_token 和 doc_token。

### 云空间文件追踪层级

```
L0  不在 InfoOps 中（排除的 / 不关注的）
     ↓ 首次与线程关联或被提及
L1  _index.md 里一行（低频，只登记 名字+类型+token+一句话定位）
     ↓ 满足升格条件
L2  <主题>.md 聚合追踪（一组相关文件合成一个 md，追踪主线变化）
     ↓ 单个文件特别高频
L3  <文件名>.md 独立追踪（单个高频高价值文件，独立 md）
     ↓ 极少触发
L4  <文件名>/ 独立目录（文件特别复杂，需拆子文件）
```

### 云空间升格条件

| 升格 | 条件（满足任一） |
|------|----------------|
| L0→L1 | 文件被线程提及、或与工作事务相关 |
| L1→L2 | 同主题 ≥2 个文件被关联、或文件开始有协作动态 |
| L1→L3 | 单个文件关联 ≥2 线程、或多人高频协作编辑、手动标记 |
| L2→L3 | 聚合 md 中某文件的变更日志明显多于其他文件，拆出独立追踪 |
| L3→L4 | 单文件追踪内容超 200 行（极少触发） |

### 云空间降格条件

| 当前 | 降格条件 | 降到 |
|------|---------|------|
| L3 独立追踪 | 所有关联线程关闭且 90 天无新协作 | L1（_index 一行） |
| L2 聚合追踪 | 主题下所有文件 90 天无变更 | L1（相关文件各保留 _index 一行） |
| L1 索引条目 | 180 天无任何线程关联 | L0（从 _index 中移除） |

### _index.md 的定位

每个类型目录（docs/ sheets/ bitables/ pdfs/）下有一个 `_index.md`，作为低频文件的统一索引：

- 每行只写：名字 | 类型 | token | 一句话定位
- 不写变更日志、不写协作者、不写关注点
- 只有值得说的文件才登记——完全不关注的文件不进 _index
- 文件升格为聚合或独立追踪后，从 _index 中移除对应行

### 聚合追踪的定位

一个聚合追踪 md 管一组相关文件（同项目、同主题、同流程）：

- 以主题命名，不以单个文件命名
- 列出主线文件表（哪几个文件、各自角色）
- 关注点写在主线文件层面，不是每个文件各写一份
- 变更日志混合记录所有文件的变化，按时间倒序
- 当某个文件的变更明显多于其他文件时，考虑拆出为独立追踪

### 云空间追踪文件的核心字段

每个被追踪的云空间文件（L2 聚合或 L3 独立），其 InfoOps 追踪文件必须包含：

```markdown
> 类型: {Doc/Sheet/Bitable/PDF} | token: <文件token> | 最后同步: <日期>
```

token 是唯一标识，文件改名不影响追踪。

## 3. 排除规则

以下类型的信息**不纳入 InfoOps 体系**，不建追踪文件，不出现在 PULSE 中：

### 明确排除

| 排除类型 | 判断依据 | 示例 |
|---------|---------|------|
| 书籍本体 | 完整出版物、ISBN 可查 | 《xxx技术指南》PDF |
| 论文本体 | 学术论文、DOI 可查 | arXiv/IEEE 论文 PDF |
| 工具/软件包 | 可执行文件、安装包、SDK 包 | .exe .dmg .tar.gz .whl .rpm |
| 安装包/分发物 | 打包分发用途的文件 | 镜像文件、补丁包 |
| 个人非工作内容 | 与工作职责无关的个人文件 | 私人照片、个人笔记 |

### 排除判断规则

1. **内容性质判断**：文件的主要用途是"被阅读/学习"还是"被使用/协作"？
   - 被阅读/学习（书籍、论文）→ 排除
   - 被使用/协作（工作文档、表格）→ 纳入

2. **来源判断**：文件是"产出物"还是"参考物"？
   - 产出物（自己或团队创建/编辑的工作文档）→ 纳入
   - 参考物（外部下载的、别人分享的阅读材料）→ 排除

3. **协作判断**：文件是否有多人协作编辑？
   - 有协作 → 即使是参考物也纳入（因为协作本身是工作信息）
   - 无协作的参考物 → 排除

4. **边界情况**：
   - 别人分享的工作文档（如"帮我看看这个方案"）→ 纳入（有协作性质）
   - 别人分享的书籍/论文（如"推荐你看看这本"）→ 排除
   - 自己上传到云空间的工具包 → 排除
   - 记录工具使用方法的文档 → 纳入（这是工作文档，不是工具本身）

### 排除信息的处理

被排除的文件不建追踪文件，但如果它们在线程中被提及，可以在线程时间线中用一句话记录：

```markdown
- <MM-DD>: <人名>分享了《<书名>》PDF，与当前方案有参考价值
```

不为这本书建 sources/space/pdfs/ 追踪文件。

## 4. 标签体系

### 完整标签表

| 标签 | 含义 | 数据源类型 | 对应目录 |
|------|------|-----------|---------|
| `[Gi:<群名>]` | 内部群 | IM | sources/groups/internal/ |
| `[Ge:<群名>]` | 外部群 | IM | sources/groups/external/ |
| `[P:<人名>]` | 人 | 跨源 | sources/people/ |
| `[SD:<名称>]` | 云文档 | 空间 | sources/space/docs/ |
| `[SS:<名称>]` | 电子表格 | 空间 | sources/space/sheets/ |
| `[SB:<名称>]` | 多维表格 | 空间 | sources/space/bitables/ |
| `[SP:<名称>]` | PDF | 空间 | sources/space/pdfs/ |
| `[C:<名称>]` | 日历事件 | 日历 | 未来扩展 |

### 标签使用规则

1. 标签内名称与 ID-MAP 或追踪文件标题完全一致
2. 有独立档案的对象前加 `★`：`[P:★<人名>]`、`[Gi:★<群名>]`
3. 外部群标签 `[Ge:...]` 本身就是提醒——Agent 看到 Ge 就知道要注意措辞
4. 一个句子中可含多个标签
5. 标签只在 线程文件、PULSE、对象档案的交叉引用处 使用，不在 ID-MAP 中使用

### 为什么群标签区分 Gi/Ge

不只是分类标记，而是**操作约束的触发器**：
- Agent 在包含 `[Ge:...]` 的线程中生成回复建议时，自动应用外部群沟通规则
- Agent 在 PULSE 的 Surge 中看到 `[Ge:...]`，优先级提升（外部群舆情更敏感）

## 5. 升格与降格规则

### 统一生命周期

```
L0  不在 InfoOps 中 / 仅在 ID-MAP 中
     ↓ 首次卷入活跃线程
L1  PULSE Directory 表一行
     ↓ 满足升格条件
L2  sources/ 下独立文件（<对象名>.md）
     ↓ 内容膨胀
L3  sources/ 下独立目录（<对象名>/ + 子文件）
```

### 群的升格条件

| 升格 | 条件（满足任一） |
|------|----------------|
| L1→L2 | ≥3 个关联线程 / 7天消息量 top5 / 手动标记 |
| L2→L3 | 需要 PROFILE + DIGEST + ROSTER 中 ≥2 个 |

升格为目录后的标准子文件：

```
<群名>/
├── PROFILE.md      群画像（定位、沟通规则、关键成员）
├── DIGEST.md       近期动态（30天滚动摘要）
├── ROSTER.md       人物名册（核心成员、活跃、沉默、跨群标记）
└── archive/        过期 DIGEST 季度归档
```

### 人的升格条件

| 升格 | 条件（满足任一） |
|------|----------------|
| L1→L2 | 跨 ≥3 群出现 / ≥3 个关联线程 / 关键协作者 / 手动标记 |
| L2→L3 | 跨群足迹复杂 + 历史线程 ≥10 |

升格为目录后的标准子文件：

```
<人名>/
├── PROFILE.md      人画像（角色、沟通偏好、能力域、跨群足迹）
└── archive/
    └── thread-history.md    历史线程索引
```

### 云空间文件的升格条件

见本文件第 2 节「云空间文件追踪层级」，cloud space 有独立的 L0-L4 体系。

### 线程的升格条件

| 升格 | 条件 |
|------|------|
| 文件→目录 | 单文件超 200 行，或产出 ≥2 份附属文档 |

升格后结构：

```
<线程名>/
├── THREAD.md           主时间线 + 待办
├── <附属文档1>.md      方案、纪要等
└── <附属文档2>.md
```

### 降格规则

| 当前 | 降格条件 | 降到 |
|------|---------|------|
| L2 文件 | 所有关联线程关闭且 90 天无新线程 | L1（PULSE 一行） |
| L1 索引 | 180 天无任何线程关联 | L0（仅 ID-MAP） |

### 存档规则

| 对象 | 触发 | 归档位置 |
|------|------|---------|
| 活跃线程关闭 | 所有待办完成 | threads/closed/ |
| 关闭线程 90天+ | 时间 | threads/closed/archive/ 或压缩为一行 |
| 群 DIGEST | 每季度 | sources/groups/.../archive/ |
| 人档案历史 | closed 线程引用 >10 | sources/people/.../archive/ |
| PULSE 快照 | 每周 | archive/snapshots/ |

## 6. 跨群人物识别

### 识别流程

```
拉取各重点群成员列表 → 提取 open_id
  ↓
同一 open_id 出现在多个群 → 标记为跨群人物
  ↓
在各群 ROSTER.md 的「跨群出现」列标注
  ↓
跨群 ≥3 或行为模式值得追踪 → 升格为 people/ 独立档案
  ↓
独立档案中维护「跨群足迹」表
```

### 跨源捕捉

人的信息不只来自群消息，还来自单聊。Agent 应综合以下渠道：

- **群消息**：某人在多个群中的发言和行为
- **单聊**：与某人的直接对话
- **云空间**：某人在文档/表格中的协作编辑记录
- **@提及**：在消息中被 @ 的频率和上下文

同一 open_id 是跨源关联的锚点。

### 跨群足迹的价值

同一人在不同群的角色和行为可能完全不同：

| 维度 | 内部群 | 外部群 |
|------|--------|--------|
| 身份 | <内部职位> | <外部公开身份> |
| 行为 | 方案评审、只在被@时回复 | 主动解答用户问题 |
| 活跃度 | 中 | 高 |
| 信息边界 | 知道内部排期 | 不暴露内部信息 |

这些差异必须记录在人物档案的跨群足迹表中。

### 外部人物的特殊处理

外部联系人（非本企业员工）的档案：
- 不写推测性信息（猜测对方公司/职位等），只写已确认的
- 标注信息来源（"在群里自我介绍为..."、"飞书名片显示..."）
- 跨群足迹只追踪与自己有交集的群

## 7. ID-MAP 分区规范

ID-MAP.md 按以下分区组织，顺序固定，不可合并或调换：

```markdown
## Internal Groups
| 群名 | chat_id | 7d消息数 |

## External Groups
| 群名 | chat_id | 7d消息数 |

## Internal Chats
| 姓名 | chat_id | 30d消息数 |

## External Chats
| 姓名 | chat_id | 30d消息数 |

## Bots
| 名称 | chat_id |

## Self
| 字段 | 值 |
```

### 铁律（不可违反）

1. 只写 ID 映射，禁止备注、总结、评价
2. 数值列（消息数）只写可机器统计的客观数据
3. Agent 不得自动向 ID-MAP 添加任何主观内容
4. 除非用户手动要求，不增加额外列
5. 新增条目追加到表格末尾，不排序
