# 信号节奏模型

InfoOps 从所有数据源采集的信号，归入五种节奏。PULSE 按此分区展示，Agent 据此判断优先级和行动。

---

## 五种节奏

| 节奏 | 图标 | 含义 | 核心特征 |
|------|------|------|---------|
| Surge | 🔥 | 短时间密集爆发 | 异常峰值，需要立即关注 |
| Active | ⚡ | 近期有进展，需跟进 | 有待办、有变化、有人等回复 |
| Ongoing | 🌊 | 长期持续，周期性 | 没有终点，需定期检查 |
| Emerging | 🌱 | 新出现，尚未成型 | 首次出现，需判断定位 |
| Cooling | 🧊 | 曾活跃但正在降温 | 可能需要干预或关闭 |

---

## 判定依据

### Surge 🔥

| 数据源 | 判定条件 |
|--------|---------|
| IM 群消息 | 24h 消息量 ≥ 日均值 3 倍 |
| IM 单聊 | 同一人 24h 内发 ≥5 条未回复消息 |
| 云文档 | 24h 内被 ≥3 人编辑，或同一人 ≥5 次编辑 |
| 电子表格 | 关键数据列出现大幅变更 |
| 多维表格 | 24h 内新增记录 ≥ 日均值 3 倍 |
| 跨源组合 | 同一线程在 ≥2 个数据源同时出现异常 |

### Active ⚡

| 判定条件 |
|---------|
| 7 天内有事件且待办未清 |
| 有人等待回复（单聊/群内 @ ）|
| 有决策卡在自己这里 |
| 云文档有未回复的评论 |
| 线程状态为 🔴 或 🟡 |

### Ongoing 🌊

| 判定条件 |
|---------|
| 持续 30+ 天且仍有周期性活动 |
| 没有明确终结时间（日常运营、周期性汇报） |
| 活动频率稳定（不是在爆发也不是在降温） |

### Emerging 🌱

| 判定条件 |
|---------|
| 首次出现：新群被拉入、新文档被分享、新人开始活跃 |
| 近 7 天内新建的对象或线程 |
| 已有对象出现新类型的行为模式 |

### Cooling 🧊

| 判定条件 |
|---------|
| 曾 Active/Surge 但 14+ 天无动静 |
| 群消息量从日活降到周活或更低 |
| 协作文档 14+ 天无编辑 |
| 线程待办无人推进且无人提起 |

## 云空间分层信号判定

不同追踪层级的文件，信号判定规则不同：

### _index 层（L1）

_index 中的文件只做存在性登记，不做信号判定。但以下情况触发升格检查：

| 触发条件 | 行动 |
|---------|------|
| _index 中的文件出现在线程的涉及列表中 | 检查是否需要升格为聚合或独立追踪 |
| 用户主动询问某个 _index 文件的情况 | 临时拉取详情，考虑是否升格 |
| 全量刷新时发现 _index 文件有密集变更 | 标记为 Emerging，建议升格 |

### 聚合追踪层（L2）

聚合追踪以主题为单位判定信号，不对单个文件判定：

| 判定条件 | 节奏 |
|---------|------|
| 主线文件中任一文件 24h 内被 ≥3 人编辑 | Surge |
| 主线文件 7 天内有变更且关联活跃线程 | Active |
| 主题关联的所有线程都已关闭，但文件仍有周期性更新 | Ongoing |
| 14+ 天无任何主线文件变更 | Cooling |
| 某单个文件变更明显多于其他文件 | 考虑拆出为独立追踪 |

### 独立追踪层（L3）

独立追踪文件按标准信号判定规则（与 IM 对象相同的 Surge/Active/Ongoing/Cooling 条件），额外增加：

| 判定条件 | 节奏 |
|---------|------|
| 文件被多人评论且有未回复评论 | Active |
| 文件权限发生变化（新增/移除协作者） | Emerging |
| 文件被移动或重命名 | Emerging |

## 节奏流转

```
🌱 Emerging
  │
  ├─ 7天后仍有活动 ──→ ⚡ Active
  │                        │
  │                        ├─ 短时爆发 ──→ 🔥 Surge ──→ 回落后恢复 ⚡ Active
  │                        │
  │                        ├─ 持续30+天 ──→ 🌊 Ongoing
  │                        │                    │
  │                        │                    └─ 14+天无动静 ──→ 🧊 Cooling
  │                        │
  │                        └─ 14+天无动静 ──→ 🧊 Cooling
  │
  └─ 7天后无活动 ──→ 🧊 Cooling
                        │
                        ├─ 重新活跃 ──→ ⚡ Active
                        │
                        └─ 30+天无动静 ──→ 关闭 / 归档
```

### 流转记录

节奏变化不写在信号对象的文件里，只反映在 PULSE.md 中。PULSE 每次刷新时重新判定所有对象的节奏。

## 跨源信号关联

### 自动关联规则

| 信号组合 | 判断 | 行动 |
|---------|------|------|
| 同一线程涉及的群 + 人 + 文档，任一 Surge | 整条线程提升关注 | 检查其他涉及对象是否也有异常 |
| 同一人 24h 内在 ≥2 群讨论同一话题 | 跨群热点 | 标记在人档案 + 相关群 DIGEST |
| 文档变更 + 群讨论同时段发生 | 可能指向同一线程 | 检查是否需要关联或开新线程 |
| 多个人在 24h 内编辑同一文档 | 协作爆发 | 检查是否有紧急任务驱动 |
| 群讨论提到某文档 + 该文档同日有变更 | 强关联 | 在线程中记录两者的因果关系 |
| 新群被创建 + 新文档被分享 + 涉及同一批人 | 新项目/新任务 | 建议开新线程 |

### 关联的展示方式

在 PULSE 的 Surge 区域，关联信号应合并展示：

```markdown
## 🔥 Surge

### <线程标题>（跨源关联）
- IM: `[Gi:<群名>]` <话题>讨论爆发（昨日 <N> 条，日均 <M> 条）
- IM: `[P:<人名>]` 单聊提到"<关键信息>"
- 空间: `[SD:<文档名>]` 被 <N> 人密集编辑
→ 判断：这些信号指向同一件事 — <综合判断>
→ 建议：<具体行动>
```

## PULSE 分区与信号节奏的对应

PULSE.md 的分区顺序固定，与信号节奏一一对应：

```
🔥 Surge       ← 最紧急，最先展示
⚡ Active       ← 需要跟进
🌊 Ongoing     ← 定期检查即可
🌱 Emerging    ← 需要判断，不需要行动
🧊 Cooling     ← 需要决策：干预还是关闭
Directory      ← 对象索引（不是信号，是参考）
Closed Threads ← 已完结（不是信号，是记录）
```

### 各分区内的数据源排列

Surge 和 Active 分区内部按数据源类型分 H3 子区：

```markdown
## 🔥 Surge

### IM
...

### 空间
...
```

Ongoing / Emerging / Cooling 分区用统一表格，不分子区（因为条目通常较少）。

## 增量刷新降级规则

当无法完成全量 PULSE 刷新时（接口不可用、凭据过期、超时等），按以下规则降级：

### 降级策略

| 场景 | 降级行为 |
|------|---------|
| 单个数据源不可用 | 保留该数据源上次刷新结果，标注"未刷新"和原因 |
| 多个数据源不可用 | 只更新可用的数据源，PULSE header 标注"部分刷新" |
| 所有数据源不可用 | 不更新 PULSE，标注"刷新失败"和时间 |
| 单个群/文件采集失败 | 跳过该对象，保留上次结果，在 PULSE 中标注 |

### PULSE header 标注格式

```markdown
# PULSE
> 最后刷新: <日期> by Agent | ⚠️ 部分刷新：<数据源1>不可用（<原因>）
```

### 陈旧数据的处理

| 陈旧程度 | 处理 |
|---------|------|
| 1-3 天 | 保留原位，标注上次成功刷新时间 |
| 3-7 天 | 信号节奏标注"⚠️ 数据陈旧"，不做升级判定 |
| 7+ 天 | 移入 Cooling 或从 PULSE 中移除，等待下次成功刷新恢复 |

### 跨源关联的降级

当部分数据源缺失时，跨源关联判定降级：
- 只有单源信号 → 不做跨源关联判定，仅记录单源信号
- 关联的数据源之一缺失 → 保留已有关联，标注"待验证"

## 信号阈值的校准

上述阈值（3倍、14天、30天等）是初始默认值。运行一段时间后，根据实际数据校准：

| 参数 | 默认值 | 校准依据 |
|------|--------|---------|
| Surge 倍数 | 3x | 如果频繁误报则调高，漏报则调低 |
| Active→Cooling 天数 | 14天 | 根据实际工作节奏调整 |
| Emerging→Active 天数 | 7天 | 根据新事务的典型成熟速度 |
| Cooling→关闭 天数 | 30天 | 根据线程的典型生命周期 |
| 日均基线计算周期 | 近 30 天 | 排除节假日和异常值 |

校准后的参数写入 `_RULES.md`，覆盖默认值。
