# 表达修正 Skill

检测并修正中文工作文档中"不像人话"的表达。适用于周报、项目汇报、进展总结、工作邮件等需要给上级或跨部门读者看的文档。

## 核心判断标准

一句话测试：**如果把这份文档投屏让领导直接念，他念到这句会不会停下来？**

会停下来的就是问题。不会停下来但读完觉得"这人文笔一般"的是次要问题。

## 检测框架：三层扫描

按严重程度从高到低扫描。第一层问题必须全部修掉，第二层尽量修，第三层视篇幅和时间决定。

### 第一层：出戏检测（最严重）

读者意识到"这不是写给我看的"，信任感崩塌。包含两个子类：

**1.1 元语言泄露** — 文本在讨论"怎么写"而不是"写什么"

检测方法：全文搜索以下关键词，命中即高度可疑：

```
写成、写为、表述、措辞、口径、上报、汇报方式、报告中、
周报、日报、文稿、改为、原文、修改为、提法、说法
```

典型症状与修正：

| 原文 | 问题 | 修正 |
|------|------|------|
| 不能把"有进展"**写成**"已闭环" | "写成"暴露改稿动作 | 不能把"有进展"**等同于**"已闭环" |
| 后续**周报**不能只报样板名 | "周报"暴露写稿讨论 | 后续**跟踪**不能只停留在样板名 |
| 不再出现"已上云"**式过满表述** | 在讨论文风 | 不再出现"已上云"**等超前结论** |
| 把平台**表述压回**"样板入口+…" | 在改稿 | 把平台**定位收回到**"样板入口+…" |
| 不再混用一个口径**上报** | 在讨论汇报方式 | 不再混在一起**跟踪** |

根因：写的人没有完成从"编辑视角"到"读者视角"的切换。

**1.2 来源暴露** — 直接引用了信息来源，读者不应该知道信息从哪来

检测关键词：

```
会议纪要、纪要中提到、根据XX的邮件、XX说过、
聊天记录显示、群里说、领导提到过
```

| 原文 | 修正 |
|------|------|
| **会议纪要**已经把路径讲清楚 | 路径已经明确 |
| 根据**XX的邮件**，方案已确认 | 方案已确认 |

### 第二层：拧巴检测（中等严重）

读者不会出戏，但会觉得"这句话读着别扭"。包含两个子类：

**2.1 搭配不当** — 动词和宾语硬拼

检测方法：提取"动词+宾语"组合，去掉主语和修饰，看这个两字组合日常说不说。

| 原文 | 测试 | 修正 |
|------|------|------|
| 本周**更能确认**的是 | "更能确认"日常说吗？不说 | 本周**可以确认**的是 |
| 围绕样板入口**收敛承接能力** | "收敛能力"日常说吗？不说 | 围绕样板入口**明确承接方向** |
| 度量闭环**未收紧** | "收紧闭环"日常说吗？不说 | 度量方式**尚未明确** |

高频万能动词黑名单（同一篇出现超过 2 次即可疑）：

```
打通、拉齐、收敛、收紧、闭环、对齐、赋能、沉淀、
拉通、抓手、颗粒度、触达、透传、穿透、降本增效
```

**2.2 语气失当** — 说教、辩解、下指令

| 类型 | 原文 | 修正 |
|------|------|------|
| 说教 | **与其继续堆能力名词，不如**先围绕… | **下一步围绕**多维分析… |
| 辩解 | 当前不应把重心放在"**为什么还没上云**" | 路径已经明确，当前重心是… |
| 下指令 | 再判断**云端承载条件** | 再判断**何时具备云端编排条件** |

语气检测关键词：

```
说教：与其…不如…、不应该、不能再、必须认识到
辩解：不是因为…而是因为…、并非…、不应把重心放在
下指令：务必、确保、要求各方、需要立即
```

### 第三层：疲劳检测（次要）

读者不会别扭，但读完会觉得"这篇东西好长好累"。包含两个子类：

**3.1 句式重复** — 同一句式骨架反复出现

检测方法：提取每句话的骨架结构，统计重复次数。

高频重复骨架黑名单（全文超过 3 次即应削减）：

```
不是X，而是Y
从A到B
不能只X，要Y
围绕X，做到Y
关键在于Z
核心是X
一是…二是…三是…
```

| 原文 | 修正策略 |
|------|----------|
| 本周最重要的变化**不是**新增了几个AI功能，**而是**管理要求… | 本周**真正的变化在于**管理要求… |
| 真正缺的**不是**表格动作，**而是**很多任务还没把… | 真正缺的**是**真实业务流… |

处理原则：保留语义上确实需要对比转折的位置，削减纯习惯性套用的位置。

**3.2 信息过载** — 一句话塞太多东西

检测方法：数顿号和并列逗号，一句内并列成分超过 4 个即标记。

| 原文 | 问题 | 修正策略 |
|------|------|----------|
| 业务入口、输入质量、人工复核、监控度量和承载条件还没有一起闭合 | 5个并列名词 | 按逻辑分组：前端（业务入口、输入质量）和后端（人工复核、监控度量、承载条件） |
| 围绕…三个样板收紧业务入口、数据来源、人工复核和度量边界 | 3样板名+4宾语=7名词 | 加"逐一"断句 |

信息密度修正策略：
- 并列 ≤4 个：保持原样
- 并列 5-6 个：用分号或"以及"分组
- 并列 ≥7 个：拆成两句或用层级结构

## 额外检测项：过度判断

工作文档中常见的"把话说满"问题，单独列出。

**过度判断关键词**（出现时检查是否有足够依据）：

```
已完成、已闭环、已打通、已上线、已落地、全面完成、
圆满完成、成功实现、显著提升、大幅改善、取得突破
```

检测原则：如果事项的实际状态是"有进展"或"部分完成"，不能用以上词汇。读者宁可看到保守但准确的判断，也不想看到事后被打脸的满分结论。

| 过度判断 | 实际状态 | 修正 |
|----------|----------|------|
| 客户端导入**已完成** | 只完成了配置，未验证 | 客户端导入配置已完成，**待验证** |
| AI能力**已全面上线** | 3个功能上了2个 | AI能力已上线2项，剩余1项预计下周 |
| 度量体系**已闭环** | 有了指标但没跑数据 | 度量指标已定义，**数据采集待启动** |

## 修正工作流

### 接收文本后的处理步骤

1. **通读全文**，标记所有读着别扭的位置（不急着改）
2. **第一层扫描**：用关键词黑名单搜索元语言和来源暴露，逐条标记
3. **第二层扫描**：检查动宾搭配和语气问题
4. **第三层扫描**：统计句式骨架重复次数，标记信息过载句
5. **额外扫描**：检查过度判断
6. **分类汇总**：按六大类列出所有问题，标注原文位置
7. **逐条修正**：从第一层开始改，改完验证不引入新问题
8. **全文通读**：改完后再读一遍，确认语感通顺

### 输出格式

修正结果分两部分输出：

**第一部分：问题清单**（让用户知道改了什么、为什么改）

按类别列出，每条包含：原文 → 问题说明 → 修正后文本。

**第二部分：修正后全文**

直接输出可用的完整文本，用户可以直接复制使用。

### 修正原则

- **最小化修改**：只改有问题的部分，不重写没问题的句子
- **保持信息完整**：不删减事实和数据，只改表达方式
- **保留作者风格**：不把所有句子统一成一种风格，保留原文的个性化表达
- **不引入新问题**：每次修正后检查是否引入了新的搭配不当或句式重复
- **宁保守不激进**：拿不准的不改，标记出来让用户决定

## 三个根因的自检

所有表达问题归根结底来自三个根因。修正时对照检查：

1. **写稿意识溢出**：写着写着开始讨论怎么写而不是写什么。对策：每句话问"领导念出来会不会别扭"
2. **句式惯性**：找到一种好用的句式就反复套用。对策：统计骨架重复次数，超过 3 次就替换
3. **动词偷懒**：不挑准确的动词，用万能动词硬搭所有宾语。对策：提取动宾组合，用"日常说不说"测试

## 与其他 skill 的关系

- 本 skill 聚焦于**工作文档的表达准确性**，不涉及 AI 痕迹去除（那是 humanizer 的范畴）
- 如果文本同时需要去 AI 痕迹和修表达，建议先用 humanizer 处理 AI 痕迹，再用本 skill 修正表达问题
- 如果用户在写周报过程中要求检查表达，本 skill 可与 project-report skill 配合使用
