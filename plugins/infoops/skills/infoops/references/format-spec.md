# 文件格式规范

所有 InfoOps 文件的格式约束、书写规则和自检清单。模板见 `templates.md`，数据源分类见 `taxonomy.md`。

---

## 1. 通用规范

### 文件命名

- 全部小写，英文，短横线分词：`<线程名>.md`、`<群名>.md`
- 人名用拼音无分隔：`<人名拼音>.md`
- 云空间文件追踪：用文件内容关键词命名，不用 token
- 禁止：空格、中文、下划线、驼峰、特殊字符
- 3 词以内，能识别即可

### Markdown 层级

- H1（`#`）：文件标题，每文件仅一个
- H2（`##`）：一级分区
- H3（`###`）：子项（时间线日期、PULSE 数据源子区等）
- H4+：避免使用，层级过深应拆文件

### 引用块

`>` 仅用于文件标题下方的元信息行：

```markdown
# <线程标题>
> 状态: 🔴 推进中 | 优先级: 高
```

不用引用块写正文。

### 表格

- 列数 ≤6，超过则拆表或改列表
- 每格 ≤15 字，详细内容用段落
- 空值写 `—`

### 时间格式

- 事件记录：`MM-DD`（跨年写 `YYYY-MM-DD`）
- 元信息：`YYYY-MM-DD HH:MM`
- 时间线倒序（最新在前），同日内正序

## 2. 标签使用规范

### 完整标签格式

```
[Gi:<群名>]    内部群
[Ge:<群名>]    外部群
[P:<人名>]     人
[SD:<名称>]    云文档
[SS:<名称>]    电子表格
[SB:<名称>]    多维表格
[SP:<名称>]    PDF
[C:<名称>]     日历事件
```

### 标签规则

1. **名称一致性**：标签内名称必须与 ID-MAP / 追踪文件标题完全一致
2. **独立档案标记**：有 sources/ 独立档案的对象前加 `★`
3. **使用位置**：线程文件、PULSE、对象档案的交叉引用处
4. **不使用位置**：ID-MAP、_RULES.md、MISSION.md（除数据源清单外）
5. **Ge 标签的隐含约束**：Agent 看到 `[Ge:...]` 自动应用外部群沟通规则

### 交叉引用方向

| 从 → 到 | 引用方式 |
|---------|---------|
| 线程 → 对象 | 标签：`[P:<人名>]` |
| 线程 → 线程 | 相对路径：`见 threads/active/<线程名>.md` |
| 对象 → 线程 | 状态 + 路径：`⚡ threads/active/<线程名>.md` |
| PULSE → 任何 | 路径：`→ threads/active/<线程名>.md` |

核心原则：**线程写事件，对象写画像，两者通过标签和路径互指，永远不重复内容。**

## 3. ID-MAP.md 格式

### 铁律

1. 只写 ID 映射，禁止备注/总结/评价
2. 数值列只写客观可统计数据
3. Agent 不得自动添加主观内容
4. 新增条目追加到末尾，不排序

### 固定分区和列

```markdown
## Internal Groups
| 群名 | chat_id | 7d消息数 |

## External Groups
| 群名 | chat_id | 7d消息数 |

## Internal Chats
| 姓名 | chat_id | 30d消息数 |

## External Chats
| 姓名 | chat_id | 30d消息数 |

## Bots
| 名称 | chat_id |

## Self
| 字段 | 值 |
```

分区顺序固定，不可合并或调换。

## 4. PULSE.md 格式

### 性质

PULSE 是 Agent 生成物，每次刷新全量重写，不手动编辑。

### 必须包含的时间戳

```markdown
# PULSE
> 最后刷新: <YYYY-MM-DD HH:MM> by <Agent/手动>
```

### 分区顺序（固定）

```
🔥 Surge
⚡ Active
🌊 Ongoing
🌱 Emerging
🧊 Cooling
Directory
Closed Threads
```

空分区保留标题，写 `（无）` 占位。

### 各分区格式

**🔥 Surge / 🌱 Emerging**：列表格式，按数据源分 H3 子区（IM / 空间）。每条含：异常描述 + 关联线程路径。Surge 必须有行动建议（`→ 建议：...`）。跨源关联信号合并为一个条目。

**⚡ Active**：按数据源分表。

Threads 表列：`线程 | 下一步 | 卡在谁 | 期限`

空间表列：`文件 | 类型 | 动态 | 需要我做什么`

IM 表列：`对象 | 动态 | 需要我做什么`

**🌊 Ongoing**：统一表格，列：`对象 | 类型 | 周期 | 上次检查 | 下次检查`

**🧊 Cooling**：统一表格，列：`对象 | 类型 | 上次活跃 | 静默天数 | 建议`

**Directory**：分两个子表。

Profiled（有独立档案）列：`对象 | 类型 | 信息源 | 活跃线程数 | 文件路径`

Tracked（仅索引）列：`对象 | 类型 | 信息源 | 关联线程`

类型列写具体类型：InternalGroup / ExternalGroup / Person / Doc / Sheet / Bitable / PDF

**Closed Threads**：表格，列：`线程 | 关闭时间 | 结论摘要 | 文件路径`。最多 20 条。

## 5. 线程文件格式

### 文件头

```markdown
# <线程标题>
> 状态: {🔴/🟡/🟢} <描述> | 优先级: 高/中/低
```

### 必须分区（顺序固定）

```
## 涉及
## 待办
## 时间线
## 备注（可选，无内容则省略）
```

### 涉及列表

```markdown
## 涉及
- `[Gi:★<群名>]` — <角色描述>
- `[Ge:<群名>]` — <角色描述>
- `[P:★<人名>]` — <角色描述>
- `[SD:<文档名>]` — <角色描述>
- `[SS:<表格名>]` — <角色描述>
```

每行：`- 标签 — 角色描述`（全角破折号 `—`）

### 待办

```markdown
## 待办
- [ ] <具体动作> → <责任人> → <期限>
- [x] ~~<已完成>~~ (<完成日期>)
```

- 已完成：`- [x]` + 删除线 + 完成日期
- 每条含具体动作 + 责任人 + 期限（如有）
- 上限 10 条，超过应拆分线程

### 时间线

```markdown
## 时间线
### <MM-DD>
- <事件>

### <MM-DD>
- <事件>
```

- H3 为日期，倒序排列
- 每条一句话，不写长段
- 人名直接写，不用标签（涉及列表已定义关系）
- 只记有意义的事件，不记流水账

### 已关闭线程

保留三个分区：`结论` + `关键决策` + `涉及`。删除时间线和待办。

## 6. 群档案格式

### 文件头

```markdown
# <群名称>
> <内部/外部>群 | 群主: <谁> | 成员: ~<N>
```

### 简单版（单文件）必须分区

```
## 定位         一句话用途
## 沟通规则     发言注意事项（外部群必须有外部注意事项）
## 关键成员     表格：人 | 角色 | 备注
## 线程索引     状态标记 + 路径列表
## 备注         可选
```

### 升格后子文件

**PROFILE.md**：同简单版，去掉线程索引。

**DIGEST.md**：

```markdown
# <群名> · 近期动态
> 覆盖: <起始> ~ <结束> | 更新: <日期>
```

分区：`摘要`（2-3句）→ `关键讨论`（H3日期+话题）→ `趋势`（活跃度/热点/新活跃/沉默）→ `待关注`

DIGEST 写**提炼后的判断**，不是聊天记录：

```
✗ <人名> 14:32 说："我觉得这个兼容性有问题"
✓ <人名>提出 <版本号> 与旧版本存在兼容性风险
```

**ROSTER.md**：三个固定表格：

```
## 核心成员     人 | 群内角色 | 跨群出现 | 独立档案
## 近期活跃     人 | 最近行为 | 值得关注
## 沉默但重要   人 | 重要原因 | 备注
```

### 外部群的额外格式要求

PROFILE.md 的沟通规则必须包含：

```markdown
## 沟通规则
- ⚠️ 外部群，以下规则必须遵守：
- 禁止暴露内部排期和未发布功能
- <其他外部群特有规则>
```

DIGEST.md 中涉及内部信息的讨论，标注 `[内部敏感]`：

```markdown
### <MM-DD>: <话题>
- <描述>
- [内部敏感] 涉及未发布特性，<人名>未提及 ✅
```

## 7. 人档案格式

### 文件头

```markdown
# <姓名>
> <角色/职位> | <部门> | 合作频率: 高/中/低
```

### 必须分区

```
## 角色关系       一段话描述工作关系
## 沟通偏好       列表
## 能力域         一句话或短列表
## 跨群足迹       表格：群 | 角色 | 活跃度 | 典型行为
## 线程索引       状态 + 路径 + 角色说明
## 长期备注       可选，按月记录
```

### 跨群足迹表

```markdown
## 跨群足迹
| 群 | 角色 | 活跃度 | 典型行为 |
|---|---|---|---|
| [Gi:<群名>] | <角色> | 高/中/低 | <行为模式> |
| [Ge:<群名>] | <角色> | 高/中/低 | <行为模式> |
```

- 活跃度只写 高/中/低
- 典型行为写行为模式，不写具体事件

### 外部人物的额外要求

```markdown
## 识别备注
- 信息来源：飞书名片 / 群内自我介绍 / 推测（标注）
- 外部群身份：<公开身份>
- 内部群身份：<如有交集>
- 信息边界确认：外部群发言是否涉及内部信息 ✅/⚠️
```

### 画像 vs 事件的边界

| 写入人档案（画像） | 写入线程（事件） |
|-------------------|----------------|
| 偏好直接沟通 | 今天在群里提出 concern |
| 擅长性能评估 | 完成了性能测试报告 |
| 周五下午经常不在线 | 周五没回复消息 |
| 对某项目持谨慎态度 | <MM-DD> 说兼容性有问题 |

## 8. 云空间追踪文件格式

### _index.md 格式

每个类型目录下的低频文件索引。

```markdown
# <类型> 文件索引

| 文件名 | 类型 | token | 定位 |
|--------|------|-------|------|
| <文件名> | Doc | <token> | <一句话定位> |
| <文件名> | Doc (Wiki) | <token> | <一句话定位> |
```

规则：
- 每行只写四列，不写变更日志、协作者、关注点
- 按登记时间追加到末尾，不排序
- 文件升格为聚合或独立追踪后，从 _index 中移除对应行
- 完全不关注的文件不进 _index

### 聚合追踪 md 格式

一个 md 管一组相关文件。

```markdown
# <主题名>
> 覆盖 <N> 个文件 | 最后同步: <日期>

## 主线文件
| 文件 | 类型 | token | 角色 | 最后变更 |
|------|------|-------|------|---------|
| <文件名> | Doc | <token> | <角色描述> | <日期> |
| <文件名> | Sheet | <token> | <角色描述> | <日期> |

## 关联线程
- threads/active/<线程名>.md

## 关注点
- <检测规则1：什么变化需要通知>
- <检测规则2>

## 变更日志
- <MM-DD>: <谁>更新了<哪个文件>（<具体位置>）
```

规则：
- 以主题命名，不以单个文件命名
- 主线文件表列出所有被管理的文件及各自角色
- 关注点写在主线文件层面
- 变更日志混合记录所有文件的变化，按时间倒序
- 当某文件变更明显多于其他文件 → 考虑拆出为独立追踪

### 独立追踪 md 格式

单个高频高价值文件。

```markdown
# <文件名称>
> 类型: <Doc/Sheet/Bitable/PDF> | token: <文件token> | 最后同步: <日期>
```

来自 Wiki 的文件：

```markdown
> 类型: <Doc/Sheet> (Wiki) | wiki_token: <wiki_token> | doc_token: <doc_token> | 最后同步: <日期>
```

必须分区：

```
## 定位         一句话说清用途和追踪原因
## 关联线程     路径列表
## 关注点       Agent 应检测什么变化（不是复述内容）
## 协作者       表格：人 | 权限 | 行为模式
## 变更日志     日期倒序
```

### 关注点的写法

写**检测规则**，不是内容描述：

```
✗ 这个表格包含 Q1 的 KPI 数据
✓ Q1 达成率列：每周一检查是否有数值变化

✗ 文档第三章是技术方案
✓ 第三章技术方案：有修改时通知我，可能影响关联线程
```

### 变更日志格式

```markdown
## 变更日志
- <MM-DD>: <谁>更新了<什么>（<具体位置>）
```

- 格式：`- <MM-DD>: <谁>做了<什么变更>（<具体位置>）`
- 变更要具体到位置，不写"文档有更新"
- 不记录格式调整、错别字修正等无实质变化的编辑

### 不同文件类型的追踪侧重

| 类型 | 追踪重点 | 关注点示例 |
|------|---------|-----------|
| 云文档 Doc | 章节变更、评论、审批状态 | "方案评审状态从'待审'变化时通知" |
| 电子表格 Sheet | 关键列/行数值变化 | "收入列数值变化 ≥10% 时标记" |
| 多维表格 Bitable | 记录增删、状态字段变化 | "新增记录时通知，状态变为'已关闭'时更新线程" |
| PDF | 版本替换、新版上传 | "有新版本上传时对比变化" |

## 9. MISSION.md 格式

### 性质

用户手动维护，Agent 只读（除非用户明确要求修改）。

### 关注优先级按紧迫度排

```markdown
## 关注优先级
1. 阻塞性问题（有人等回复、决策卡在我这里）      ← 立即
2. 进行中线程的状态变化                            ← 当日
3. 重点人/群的异常信号                              ← 当日
4. 云空间文件的协作动态                             ← 次日可
5. 新出现的信号                                     ← 本周内判断
```

### 决策原则写可执行规则

```
✗ 注意沟通方式
✓ 外部群发言不暴露内部排期和未发布功能

✗ 重视技术评审
✓ 技术方案需<人名> review 后才能推进到执行阶段
```

### 数据源清单

每行四列，缺一不可：`源 | 接口 | 采集策略 | 状态`

状态：`✅ 已接入` / `🔲 待接入` / `❌ 不可用`

## 10. 升格文件结构

### 入口文件命名（大写）

| 对象 | 入口文件 | 其他子文件 |
|------|---------|-----------|
| 群 | PROFILE.md | DIGEST.md, ROSTER.md, archive/ |
| 人 | PROFILE.md | archive/ |
| 线程 | THREAD.md | <附属文档>.md |

入口文件大写，子文件小写。

### 升格 = 文件变同名目录

```
升格前: sources/groups/internal/<群名>.md
升格后: sources/groups/internal/<群名>/PROFILE.md
```

原文件内容迁移到入口文件，不保留原 .md 文件。

## 11. 书写边界与自检

### 什么该写 / 不该写

| 场景 | 该写 | 不该写 |
|------|------|--------|
| 群讨论 | 结论、暴露的问题、产生的影响 | 谁在几点说了什么（聊天记录） |
| 人物 | 行为模式、沟通偏好、能力 | 单次具体发言内容 |
| 文档 | 什么位置改了什么 | 改前改后的完整内容 |
| 线程 | 发生了什么、结论、下一步 | 讨论中的来回拉扯 |
| 待办 | 具体动作 + 责任人 + 期限 | "继续推进"、"持续关注" |

### 常见违规

```
✗ "<人名>在群里说了很多关于兼容性的问题"
✓ "<人名>提出 <版本号> 与旧版本存在兼容性风险，需要评估"

✗ "今天群里很活跃"
✓ "<群名>今日 47 条消息（日均 8 条），集中在 <话题>"

✗ "文档有更新"
✓ "<人名>更新了 <文件名> 第3列（<具体数据>）"

✗ "持续关注"
✓ "下周一检查是否有新变化，无变化则降为 Cooling"
```

### Agent 写入前自检清单

1. [ ] 写的是判断和结论，不是原始数据搬运？
2. [ ] 事件细节在线程里，画像在对象档案里，没有混放？
3. [ ] ID-MAP 没有被添加主观内容？
4. [ ] 标签名称与 ID-MAP 一致？
5. [ ] 涉及外部群 `[Ge:...]` 的内容已检查信息边界？
6. [ ] 待办有具体动作 + 责任人，不是空话？
7. [ ] 时间线倒序？
8. [ ] PULSE 的 Surge/Active 有行动建议？
9. [ ] 云空间文件的变更日志具体到位置？
10. [ ] 被排除的信息（书籍/论文/工具包）没有建追踪文件？
