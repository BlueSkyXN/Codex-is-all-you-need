# Agent Skill 开发规范

| 项 | 值 |
|---|---|
| 文档 ID | `skill-spec` |
| 版本 | 0.1 |
| 日期 | 2026-07-17 |
| 定位 | 通用 Agent Skill 开发规范：分类、结构、写法、组织、版本与协作 |
| 分发 | **自包含单文件**：执行本文不依赖任何外部文件 |
| 适用对象 | Skill 作者、评审者、AI Agent |

本规范定义 Agent Skill 的分类体系、包结构、`SKILL.md` 写法、多 skill 组织、评测与版本协作规则，适用于任何兼容 [Agent Skills 开放标准](https://agentskills.io) 的运行时（Claude Code、Codex、Copilot CLI、OpenClaw、CodeBuddy、WorkBuddy、Qoder 等）。

条款以三个来源校准，不设立缺乏依据的约定：

1. **开放标准**：Agent Skills 开放标准定义的可移植底座。
2. **平台硬限**：各运行时公开文档中的强制约束。
3. **工程实践**：多平台生产环境 skill 仓库的既有共识。

---

## 1. 如何使用本规范

### 1.1 读者路径

| 你要做的事 | 阅读 |
|---|---|
| 30 秒了解全貌 | 附录 A 一句话备忘 |
| 新建一个 skill | 第 12 章模板起步 + 第 2～6 章 + 第 11 章清单 |
| 个人自用、暂不发布 | 第 12 章模板 + 第 2～6 章；第 8～9 章与 5.4 平台发布注意可暂缓 |
| 评审 skill | 第 10 章禁止清单 + 第 11 章合格清单 |
| 组织多个 skill / 建 router | 第 3、7 章 |
| 定版本、多人协作 | 第 9 章 |
| 生成或校验 skill（AI Agent） | 1.2 节 + 全部「必须 / 禁止」条款 |

本规范分两级适用：**可移植合规**（第 2～6 章与第 10～11 章清单），任何 skill 都适用；**协作治理**（第 9 章与 6.1 治理层字段），多人协作或公开发布时为必须，纯个人自用降为建议。

### 1.2 Agent 执行规则

AI Agent 依据本规范生成或评审 skill 时：

1. 「必须 / 禁止」是硬约束；「建议」默认遵循，偏离时说明理由。
2. 目录结构优先使用第 5 章标准目录；新增顶层目录必须能陈述标准目录覆盖不了的独立职责。
3. 结构按需生长：先判断现有材料单文件是否容纳得下，再决定建目录。
4. 条款冲突时：更具体的条款优先于总述性原则。
5. 「说明 / 理由」性文字不构成新目录或新例外的授权。

### 1.3 用语强度

| 用语 | 含义 |
|---|---|
| **必须** / **禁止** | 违反即不合格 |
| **建议** | 默认遵循；偏离需能说明理由 |
| **可以** | 可选 |

### 1.4 范围

- 本规范管：单个 skill 包与多 skill 组织的设计、结构、写法、评测、版本与协作。
- 本规范不管：Agent / 角色系统设计、中央知识库选型与权限模型、各运行时的内部实现细节。

## 2. 核心模型

### 2.1 Skill 的定义

**Skill 是可触发的工序单元**（process unit）：封装一段可复用的工作流程，由运行时在匹配场景中加载执行。

- 一个 skill = 一个目录 + 一个 `name` + 一个 `SKILL.md` 入口。
- Skill 不是长文百科，不是中央知识库，不是配置开关。

### 2.2 三层加载模型（渐进披露）

Skill 内容按三层加载，各层预算不同。**本规范的全部内容落点规则都由这个模型推导：**

| 层 | 加载时机 | 承载内容 | 预算特征 |
|---|---|---|---|
| L1 发现层 | 会话启动即注入，与所有已装 skill 共享预算 | `name` + `description` | 极小，按字符计 |
| L2 执行层 | 触发后整体加载 | `SKILL.md` 正文 | 有限，应精炼 |
| L3 资源层 | 执行中按需读取 | `references/`、`examples/`、`scripts/`、`assets/` | 按需，不占触发预算 |

四条推论：

1. 触发条件**必须**写进 `description`——skill 能否被自动选中，跨运行时只看这一句（个别运行时另有私有触发字段、门控开关与显式调用，属平台扩展，不能替代 `description`）。
2. 主工作流**必须**写在正文——L2 是触发后唯一保证加载的内容。
3. 执行中才需要的细节放 L3，正文只留链接与使用时机。
4. 运行时不读的内容与超出包尺度的内容不占任何层：评测材料进 `evals/`，中央知识放包外。

### 2.3 核心原则

1. **Git 为协作真源，平台为发布渠道**：真源（source of truth）在 Git；上传、插件市场上架是发布动作，不是多人共编方式；协同、评审、合并在 Git 完成。
2. **可移植底座 + 平台适配**：以开放标准（`SKILL.md` + `name` / `description` + 渐进披露）为底座；平台私有能力通过各自的扩展落点实现（见 6.1），不污染底座。
3. **结构按需生长**：从单文件开始，材料真实出现后再建目录；**禁止**预建空目录。
4. **引用而非复制**：外部 skill 与中央知识只引用，不复制正文进包。
5. **同一事实只留一份真源**：规则写一处，其余位置引用。

## 3. 分类体系

### 3.1 形态分类

形态描述包的**现状**，不是升级路线图；一个 skill 可以永远停留在单文件形态。

| 形态 | 判定 | 结构特征 |
|---|---|---|
| **单文件**（single-file） | 只需 `SKILL.md` 即可完整执行 | 仅 `SKILL.md` |
| **标准**（standard） | 单文件容纳不下，出现配套文件 | + `references/` / `scripts/` / `examples/` 中任意 |
| **套件**（multi-route） | 一包内多条**同一意图族**路线 | 正文含路线选择；配套目录成型 |
| **群**（fleet） | 数十个独立 skill 的平级组织 | 多包平级 + 可选 router |

判定顺序：先分单包（前三种形态）与多包（群）；单包内再看文件形态与路线数。

术语提示：本规范的「套件」（multi-route）指单包多路线，不同于 Codex 生态中编排 agent 的 suite，也不同于 QoderWork 等产品中打包多个 skill 的角色 plugin——那些属于 agent 编排与分发概念，不是单个 skill 包的形态。

### 3.2 套件规则

1. **必须**仍是一个包：一个 `name`、一个 `SKILL.md`；**禁止**包内嵌套子 `SKILL.md`。
2. **必须**多条路线同属一个意图族；不同意图族的工作流拆为独立 skill。
3. 意图族（intent family）判定：多条路线能被同一条 `description` 自然覆盖、无需用「或」并列互不相干的场景，即属同一意图族；反之应拆为独立 skill。该判定源自三层加载模型——套件的全部路线共用一条发现层描述（见 2.2）。判例：「提取 PDF 文本、填写表单、合并文件，处理 PDF 时使用」属同一意图族（同一对象与场景入口，开放标准的示例 description 即此写法）；「生成周报或初始化数据库」不属（场景互不相干，只能靠「或」并列）。
4. **必须**在正文写清路线选择规则：何种输入或场景走哪条路线。

### 3.3 群规则

1. 群内每个叶子（leaf，即群内单个 skill）独立满足单包规则（第 5～6 章），平级排列。
2. Router（引导 skill）是可选件，不是标配：实践参考线约 30 个以上独立可触发 skill、且入口多样到需要引导时才值得建；叶子数量达数十时才需要两级 router。
3. 规模未达参考线时，**建议**不建 router：靠 `description` 区分 + 文档级地图（README / docs）即可；因产品入口等原因提前建的，须说明理由，且 router 仍须满足 7.2。
4. **禁止**把同一 workflow 拆成 Phase 1…N 伪装成多个独立 skill。

## 4. 内容落点

### 4.1 落点决策

判定问题：**这段材料执行时要不要被 agent 读？何时读？**

| 材料 | 落点 | 对应层 |
|---|---|---|
| 触发条件 | `description` | L1 |
| 主工作流、完成条件、硬禁令 | `SKILL.md` 正文 | L2 |
| 执行时按需查的规则、协议、契约、排查手册 | `references/`（篇幅短则并入正文） | L3 |
| 输入输出样例、正反例、长支路演示 | `examples/` | L3 |
| 可靠可复跑的命令与脚本 | `scripts/` | L3 |
| 输出模板、静态资源 | `assets/` | L3 |
| 触发 / 质量回归材料 | `evals/` | 不加载 |
| 大体量 / 多包共享 / 有权限边界的语料 | 包外（知识库、MCP（Model Context Protocol）服务、独立仓库） | 包外 |
| 外部 skill 的工序 | 包外引用：写清何时用、怎么调 | 包外 |

### 4.2 资料目录分工

```text
references = 怎么做对（执行说明）
examples   = 长什么样（示范）
evals      = 测过了吗（回归；运行时不读）
```

同一事实只留一份真源：硬性规则写在正文或 `references/`；`examples/` 只做示范；`evals/` 只写可观察断言。三者都可以没有。

### 4.3 包外原则

材料命中任一条件即放包外：

1. 体积大（数千 token 以上）。
2. 更新节奏独立于 skill 流程。
3. 被多个 skill 共享。
4. 存在权限或密级差异。

规则：

- 包内**可以**写「怎么查」：用什么服务或命令、典型查询、结果怎么用；直接写进正文或 `references/`，不需要专用配置格式。
- **禁止**把中央知识正文复制进包内；**禁止**建立包内快照目录形成第二真源（真源漂移的根源）。
- 引用外部 skill 时：用稳定名称 + 写清前置条件与失败降级；**禁止**复制对方正文。

## 5. 包结构

### 5.1 标准目录

```text
<skill-id>/                  # 目录名 = name
├── SKILL.md                 # 必须：唯一入口
├── references/              # 可选：按需读的执行说明
├── examples/                # 可选：样例与演示
├── scripts/                 # 可选：可执行辅助
├── assets/                  # 可选：模板与静态资源
├── <平台 adapter>            # 可选：按目标平台要求放置（见 6.1）
├── evals/                   # 可选：回归材料（运行时不读）
└── README.md / LICENSE*     # 可选：分发说明，不承载工序
```

目录出处：`references/`、`scripts/`、`assets/` 是开放标准点名的推荐可选目录；`examples/`、`evals/` 与平台 adapter 为本规范的工程补充，落在开放标准「允许额外文件与目录」的范围内。

结构规则：

1. **必须**：有且仅有一个 `SKILL.md`；目录名与 `name` 一致。
2. **禁止**：包内第二个 `SKILL.md`（含子目录嵌套）——那是两个 skill，应拆分。
3. **禁止**：预建空目录。
4. **禁止**：包内版本目录（`v2/`、`v3/`）；版本换代走版本契约（第 9 章）或开平级新包（`<name>-v2`）。
5. **建议**：内容优先落入标准目录；新增顶层目录须有标准目录覆盖不了的独立职责，并在评审中说明。
6. 历史包若有说明文件平铺在包根，不判不合格，但新建包不采用；引入外部包同理，标准目录之外的既有目录随大改收敛。

### 5.2 生长规则

1. 从单文件 `SKILL.md` 开始。
2. 某类材料真实出现时才建对应目录：执行说明成型 → `references/`；样例沉淀 → `examples/`；命令固化 → `scripts/`；模板复用 → `assets/`；回归需求出现 → `evals/`；面向特定平台发布 → adapter。
3. 没有新材料就停留在当前形态；形态只描述现状，不构成升级压力。

### 5.3 命名规则

**必须**：

1. 目录名 = `name`：全小写英文，词间用连字符，不含空格、大写与下划线（如 `weekly-report`、`pdf-extract`），长度不超过 64 字符（开放标准上限）；形式化定义为正则 `^[a-z0-9]+(?:-[a-z0-9]+)*$`（kebab-case）。
2. 入口文件名固定为 `SKILL.md`。

部分产品 UI 允许本地化命名（如中文 skill 名、展示名）或省略 `name` 回退目录名，均属产品私有行为，不构成可移植包的例外：跨运行时分发仍按上述规则命名，UI 展示名放平台扩展字段（见 6.1）。中文意图的命名路径：按意图取达意的英文名（「周报生成」→ `weekly-report`，「PRD 撰写」→ `prd-writer`），中文触发词写进 `description`，中文展示名放平台扩展字段——触发靠 `description` 而非 `name`（见 6.2），英文命名不影响中文场景的触发。

**建议**：

1. 群内用前缀分域（如 `dev-` / `data-` / `office-`），兼容各运行时的扁平发现模型。
2. `references/` 用语义名平铺（`output-contract.md`、`troubleshooting.md`）；同主题文件多时最多一层子目录。
3. `examples/` 用语义名；正反例以 good / bad 标注（前后缀均可）；演示序列可用 `NN-<slug>.*`；不用日期做文件名——样例是稳定资产，不是日志。
4. `scripts/` 用动宾式命名（`validate_output.py`）；公共模块命名不限。

### 5.4 发布与提交

| 内容 | 处置 |
|---|---|
| `SKILL.md`、`references/`、`examples/`、`scripts/`、`assets/`、adapter、分发说明文件 | 随包发布 |
| `evals/` | 建议随包，便于消费方回归；非运行时依赖 |
| 生成物与系统垃圾（缓存、`.DS_Store` 等） | **禁止**提交 |
| 密钥、内网地址、真实个人 / 客户数据 | **禁止**出现在包内任何文件 |

安全规则：

1. 脚本行为**必须**与正文声明一致；**禁止**执行正文未声明的网络外发、数据上传或删除性操作。
2. 正文与 `description` **禁止**指示 agent 绕过运行时的权限确认或安全机制。
3. **建议**引入第三方 skill 前通读其 `SKILL.md` 与脚本：skill 同时是提示词与可执行代码，应按不可信代码对待。

平台发布注意：各平台对文件类型与包体积有硬限（如仅接受文本文件、单包 50MB 级），发布前核对目标平台文档；打包由脚本完成，真源始终在 Git。

## 6. SKILL.md 写法

### 6.1 Frontmatter

Frontmatter 指 `SKILL.md` 开头两条 `---` 之间的元信息区（YAML 格式），运行时靠它发现与识别 skill。部分产品在 UI 中代填此区；手写包时按下式给出：

```yaml
---
name: <与目录名一致>
description: >-
  <做什么>。当 <用户真实说法的触发场景> 时使用。
metadata:
  version: "0.1"          # 两段式字符串，契约见第 9 章
  updated: "YYYY-MM-DD"
---
```

字段分三级：

| 级 | 字段 | 强度 |
|---|---|---|
| 可移植底座 | `name`、`description` | **必须**（开放标准要求） |
| 治理层 | `metadata.version`、`metadata.updated` | 多人协作或公开发布场景**必须**；纯个人自用**建议** |
| 平台扩展 | 运行时私有字段 | 仅面向该平台发布时写 |

字段规则：

1. `name`：**必须**与目录名一致。
2. `description`：**必须**；写法规则见 6.2。
3. `metadata.version`：行为版本，两段式字符串；契约见第 9 章。
4. `metadata.updated`：`YYYY-MM-DD`，取最后一次实质变更日期。
5. **可以**使用 `license`、`compatibility` 等开放标准可选字段；私有属性放 `metadata` 键下，开放标准建议客户端忽略不认识的 `metadata` 键。顶层未知字段被忽略还是拒绝，跨运行时行为未经验证，混投多平台前先做冒烟验证（装入目标平台，实测能否发现、触发与执行）。

平台扩展的常见落点（互不污染，按目标平台选用）：

| 扩展方式 | 典型平台 | 形态 |
|---|---|---|
| Sidecar 文件 | Codex | `agents/openai.yaml` |
| Frontmatter 运行时字段 | Claude Code、CodeBuddy | `allowed-tools`、`context` 等 |
| Frontmatter 厂商命名空间 | OpenClaw | `metadata.openclaw.*` |
| Frontmatter 生命周期 / UI 字段 | WorkBuddy、QoderWork | `agent_created`、双语展示名等 |

第四类是无命名空间的顶层私有字段，跨运行时兼容性最弱（见上方字段规则 5），仅面向对应产品发布时保留。

### 6.2 触发设计（description）

skill 能否被自动选中，跨运行时只取决于 `description`——它是发现层（L1）中唯一可移植的内容；运行时叠加的私有触发字段、门控与显式调用均属平台扩展，不可移植，也不改变以下规则：

1. **必须**同时包含「做什么」与「何时用」；触发信息**禁止**只写在正文。
2. 「何时用」使用用户的真实说法（关键词、动词、场景），不用抽象概括。
3. **建议**补充「何时不用」，尤其在发生过误触发之后。
4. 长度：**建议** 100～200 字；可移植上限 1024 字符；部分运行时的列表展示另有更短的截断阈值，关键信息前置。
5. **建议**语言与目标用户的会话语言一致；多语言用户场景下，关键触发词可双语并列。
6. 触发问题的修正路径：漏触发 → 补充用户真实关键词；误触发 → 收窄场景描述或补「何时不用」。

### 6.3 正文

硬要求四条：

1. 主工作流**必须**写在正文；**禁止**正文只剩链接壳。
2. 关键路线**必须**有可观察完成条件（命令、产物、检查项）。
3. 套件**必须**写清路线选择。
4. 相对链接**必须**全部可达；**禁止**预链不存在的文件。

**建议**骨架（非强制）：

```markdown
# <标题>
## 适用 / 不适用
## 前置检查
## 工作流          # 每条路线附完成条件
## 输出与完成条件
## 异常与禁止
## 参考            # 只链真实存在的文件与外部 skill
```

正文保持精炼（数百行以内），细节移入 `references/`。

## 7. 多 Skill 组织

### 7.1 布局

```text
skills/
  <router>/           # 可选引导包，本身是合法单包
  leaf-a/             # 每个叶子独立满足第 5～6 章
  leaf-b/
  …
```

平级目录 + 前缀分域；文档级地图放 README / docs，不占运行时。

### 7.2 Router 规则

1. Router **只选路**：意图识别 → 路由表 → 交给目标 skill 执行。
2. **禁止**复制叶子正文；**禁止**在用户要结果时只指路不执行。
3. 路由表**建议**包含：条件、目标 skill、选择原因；外部前置（登录态、服务可用性）单独列出。

### 7.3 Skill 间调用协议

群内一个 skill 调用另一个时，调用方正文写清：

```markdown
当 <条件>：使用 `<skill-name>`
- 传入：…
- 期望返回：…
- 返回后本 skill 继续：…
```

## 8. 评测（evals）

1. 只回答两个问题：**该不该触发**（trigger）、**结果过不过**（quality）。
2. 运行时不读；读者是作者与 CI。
3. 形态自由：json / yaml / md 均可。**建议**至少覆盖「应触发 / 不应触发」两类样本；质量断言写可观察要点与禁令，不复制正文规则。
4. 出现回归需求时再建（`description` 大改、发生过误触 / 漏触、完成条件变更）；**禁止**预建空 `evals/`；**禁止**把日常操作步骤写进 evals 替代正文。

## 9. 版本与协作

### 9.1 版本契约

1. `metadata.version` 是**两段式字符串**（`主.次`，字符串序列而非十进制数），首个实质状态为 `"0.1"`。
2. 向后兼容的实质变化升次段：`0.9 -> 0.10`、`1.0 -> 1.1`。适用：工作流增补、规则修正、验证增强、输出改进。
3. 破坏性变化升主段并归零次段：`0.10 -> 1.0`。适用：破坏公开调用名、必需输入输出契约、产物 schema、脚本 CLI、核心默认流程，或其他 skill 消费的契约。
4. **实质内容**（版本指纹范围）= 正文（不含 `version` / `updated` 两行）+ `references/` + `scripts/` + `assets/` + `examples/` + 平台 adapter。`README*`、`LICENSE*`、`evals/`、系统垃圾、纯元信息修改不算实质变更——仅评测材料变化不构成行为变更，不升版。
5. `metadata.updated` 取最后一次实质变更的日期（Git committer date，`YYYY-MM-DD`）。
6. 发布包（plugin / bundle）的版本与 skill 行为版本**分开**管理。

两段式版本、指纹范围与日期取值是**本规范的治理约定**，非开放标准要求：开放标准仅把 `metadata.version` 作为可选自由字符串；插件市场普遍另用三段式 semver，故发布包版本按第 6 条分开管理。

**建议**配 CI 门禁（gate），至少校验：目录名 = `name`；version 两段式、updated 格式合法；实质内容变更时版本与日期已同步更新。

### 9.2 协作流程

| 变更 | 流程 |
|---|---|
| 正文 / `references/` / `scripts/`（行为面） | PR + 评审 |
| `examples/` 新增样例 | 轻量评审 |
| `evals/` | 与对应行为变更同一 PR |
| 发布到平台 | 构建产物上传；真源在 Git |

若仓库维护 skill 的镜像副本（示例目录与发布包并存同一份 skill），镜像**必须**与真源逐字节一致，由门禁校验。

### 9.3 废弃与替代

1. skill 下线或被替代时，**建议**在 `description` 标注 deprecated 并指向替代 skill，保留至少一个发布周期后再移除。
2. 大版本换代采用平级新包（`<name>-v2`）时，旧包按上条处理。

## 10. 禁止清单

1. 包内第二个 `SKILL.md`；子目录嵌套 skill。
2. 预建空目录。
3. 无独立职责、与标准目录语义重叠的自造顶层目录。
4. 正文断链；预链不存在的路径。
5. 触发条件只写正文、不进 `description`。
6. 正文空心化：主工作流不在 `SKILL.md`。
7. 复制外部 skill 或中央知识正文进包；包内快照第二真源。
8. Router 复制叶子正文；用户要结果时只指路不执行。
9. 套件混入不同意图族的工作流；把同一工作流的 Phase 拆成多个 skill 虚增数量。
10. 把 evals 当运行时手册；硬性规则只写在 examples。
11. 提交生成物、系统垃圾、密钥或真实敏感数据。
12. 脚本执行正文未声明的外发或删除操作；指示 agent 绕过运行时安全确认。

## 11. 合格清单

### 11.1 所有单包

- [ ] 目录名 = `name` = kebab-case；有且仅有一个 `SKILL.md`
- [ ] 顶层目录符合 5.1（新增目录有职责说明）
- [ ] `description` 含做什么 + 何时用（触发信息在发现层）
- [ ] 主工作流在正文且有可观察完成条件
- [ ] 无断链；无空目录；无生成物 / 敏感数据
- [ ] 脚本行为与正文声明一致
- [ ] 外部依赖与中央知识只引用不复制

### 11.2 多人协作 / 公开发布追加

- [ ] `metadata.version` / `metadata.updated` 存在且合法（配了门禁则门禁通过）

### 11.3 套件追加

- [ ] 多路线同一意图族；正文有路线选择
- [ ] 无嵌套子 skill

### 11.4 群追加

- [ ] 叶子平级、各自独立合格；router（若有）只选路
- [ ] 规模未到参考线时未建 router（提前建的有理由说明）
