# 项目功能能力地图方法

这个 skill 要刻意保持轻。它不是正式项目研究包，也不是项目管理计划。它只负责让本地 AI 快速读代码、`local/` 材料和 git 记录，把一个项目整理成“能力域、功能组、功能点、证据、状态、下一步”。

## 1. 默认只产出一张能力地图

默认结构是：

```text
项目目标
-> 一级能力域
-> 二级功能组或模块
-> 三级功能点
-> 证据
-> 状态
-> 下一步
```

它要回答的问题是：

```text
这个项目做了什么？有哪些能力域？每个能力域下有哪些模块？每个模块已经形成哪些功能点？后续还应补哪些能力点？
```

除非用户明确要求，不要展开产品结构、工作分解、跟踪矩阵、成熟度、卡点长报告、规格卡、部署检查、目标变化日志或大量表格。用户明确要求 XLSX / Excel / 表格阅读版时，只额外生成一个完全汉化的 `项目能力表.xlsx`。

## 2. 层级定义

| 层级 | 含义 | 示例 |
|---|---|---|
| 项目目标 | 一句话说明项目用途 | `面向跨境电商质量分析的 AI 工作台` |
| 一级能力域 | 项目的主要能力板块 | `AI工作流/智能中台能力域` |
| 二级功能组或模块 | 能力域下的功能组、模块或子系统 | `Dify封装` |
| 三级功能点 | 用户或系统能观察到的具体功能 | `工作流调用` |

默认不展开到研发任务或正式规格。要写软件需求，用 `sdlc-srs-workflow`；要写 UI/API/Data/Admin/Permission/Directory 规格切片，用 `sdlc-spec-slice-writer`；要拆研发任务，用 `sdlc-dev-handoff-planning`。

## 3. 同时处理当前能力和后续能力

能力地图不是只列“已经完成”。当用户要求结合 git 记录、`local/` 材料或推测下一步时，要把后续 L1 / L2 / L3 也放进同一张表，但必须用状态和证据边界区分。

证据优先级一般是：

1. 可运行代码、测试、脚本、SQL/seed、配置和真实执行结果。
2. 仓库文档、接口契约、状态机、PRD、phase gate、forward plan、handoff 和验收报告。
3. git commit / merge 记录能证明的演进方向。
4. 由目录结构、命名和相邻材料推断出的方向。

状态写法：

- 已有代码并能跑通：`可运行`。
- 已有代码或配置但未实际验证：`已开发` 或 `待验证`。
- 已有设计、DDL、接口、模板或契约：`已设计`。
- 来自 roadmap / forward plan / PRD 的后续能力：`已规划`。
- 仅凭演进线判断：证据或备注必须写 `[推断]`。

不要把 `已规划`、`待验证` 或 `[推断]` 写成已经完成。

## 4. 先按业务能力拆，不按目录机械拆

优先按“项目能做什么”来拆，不要直接照搬 `frontend`、`backend`、`scripts`、`infra` 这类目录名。

工程目录仍然是证据来源，但不是能力结构本身。一个功能点可能同时由页面、接口、数据、AI 调用和配置共同支撑。

AI 或平台类项目常见的一级能力域包括：

- `前端工作台能力域`
- `后端控制面能力域`
- `AI工作流/智能中台能力域`
- `文件/OSS服务能力域`
- `数据中台/AI+BI能力域`
- `业务智能体能力域`

这些只是参考，不是必须套用。目标项目没有证据支撑的能力域不要强行写进去。

## 5. 功能点必须具体

三级功能点必须是可观察功能。好的写法：

- `工作流调用`
- `参数传递`
- `任务状态查询`
- `文件上传`
- `人工复核`
- `调用日志记录`

太虚的写法要改掉：

- `平台能力`
- `后端逻辑`
- `AI模块`
- `数据支持`
- `系统建设`

## 6. 证据规则

每个三级功能点尽量给出证据：

- 代码路径：`src/services/workflow.py`
- 页面路径：`src/pages/tasks/index.tsx`
- 接口路径：`apps/api/routes/task.py`
- 配置路径：`docker-compose.yml`
- 提示词或工作流：`prompts/monthly-report.md`
- 测试路径：`tests/test_task_api.py`
- 文档路径：`README.md`
- local 材料：`local/forward-plan/02-prd.md`
- git 记录：`git log: abc1234 commit title`

没有直接证据时：

- 用 `[推断]` 标记，并说明是从哪个目录、路由、配置或文档推断出来的。
- 用 `[未知]` 标记应该确认但没找到证据的内容。
- 不要从其他项目借能力项来补齐表格。

`local/` 常被 `.gitignore` 忽略。做这类研究时不能只看 `git status` 或普通 diff，要直接列目录、读文件，并把关键 local 材料作为事实源之一。

git 记录用于识别演进方向，不要把 commit title 过度解读成完整功能事实；必要时用 `git show --stat` 或真实文件内容复核。

## 7. 状态要简单

每个三级功能点只选一个主状态：

- `已规划`
- `已设计`
- `已开发`
- `可运行`
- `待验证`
- `已废弃/重构`
- `未发现证据`

不要使用成熟度等级、加权进度、完成百分比或汇报式结论。

## 8. 输出语言

默认产物使用简体中文：

- 文件名用中文。
- 标题、章节、说明、表头、状态、备注、下一步都用中文。
- CSV 表头也用中文。

保留原文的内容只限于真实标识，例如代码路径、命令、包名、API 名、配置键、skill 名、文件扩展名和代码标识符。

默认 CSV 表头：

```csv
一级能力域,二级功能组或模块,三级功能点,功能说明,证据路径,当前状态,下一步,备注
```

## 9. XLSX 阅读版

当用户要求 XLSX / Excel / 表格阅读版时，XLSX 只做阅读层，不改变事实源。

推荐 workbook：

- `阅读说明`
- `状态汇总`
- `能力表`
- `L1-L2索引`

其中 `能力表` 必须与 `项目能力表.csv` 同行、同列、同顺序。`状态汇总` 只做数量统计和状态解释，不引入评分、完成率或成熟度。`L1-L2索引` 只做 L1 / L2 聚合，方便浏览。

## 10. 下游承接

这份能力地图可以交给后续工作继续用：

- `sdlc-prd-workflow`：根据能力地图选择需求范围。
- `sdlc-srs-workflow`：把选中的三级功能点提升为软件需求、验收条件和 requirement IDs。
- `sdlc-spec-slice-writer`：把能力点细化为 UI/API/Data/Admin/Permission/Directory 规格切片。
- `sdlc-dev-handoff-planning`：把已经具备需求/规格或清晰 direct-dev 输入的缺口展开成研发任务和依赖顺序。
- 汇报类工作流：把能力地图整理成进展、月报或绩效材料。

优先让下游步骤去加优先级、排期、负责人、评分和汇报话术。这个 skill 不提前替人做这些判断。
