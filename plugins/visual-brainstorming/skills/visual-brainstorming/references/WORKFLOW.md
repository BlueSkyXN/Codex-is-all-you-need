# 标准工作流

本流程展开 `SKILL.md` 的快路径，描述一次完整的“识别问题 → 展示方案 → 回收选择 → 固化结论”。正常执行不需要逐字读取本文件；只有复杂会话或流程不确定时再参考。

## 阶段 1：读取任务上下文

先确认：

- 产品或技术目标；
- 受众、终端尺寸、技术栈；
- 已确定的约束；
- 当前真正未确定的事项。

不要用视觉页面代替必要的项目理解。

## 阶段 2：识别单一视觉决策

依次判断：

1. 是否涉及布局、空间、层级、流程、状态或数据流？
2. 是否存在 2–4 个结构上真实不同的方案？
3. 看图是否比文字或表格明显更清楚？

三个答案全部为“是”时才使用本 Skill；否则优先用聊天、表格、Mermaid 或单张静态图。

把宽泛问题拆成单一决策：

```text
“重做后台”
→ 导航结构
→ 信息密度
→ 筛选方式
→ 详情呈现
```

每一项分别处理，不要一次改变全部变量。

## 阶段 3：处理同意

用户已经明确要求网页、mockup 或点击选择时，可以直接继续。

隐式触发时，首次需要打开页面前说明：

- 会打开本机地址；
- 页面由 Agent 生成；
- 点击会记录到项目临时目录；
- 可以拒绝并继续只用聊天。

用户拒绝后不再重复询问。

## 阶段 4：定义本屏

先写一份最小决策说明：

```text
问题：首屏采用哪一种信息入口？
固定：功能、品牌、CTA、目标受众。
变化：首屏的信息优先级。
观察：理解速度、信任建立、进入产品的距离。
数据：全部为结构示意，无实测性能数据。
```

再为每个方案定义：

- 稳定 ID；
- 一句话核心差异；
- 适用条件；
- 主要收益；
- 主要代价；
- 可视化结构。

默认 2–3 个方案；只有第四个方案具有独立逻辑时才增加。

## 阶段 5：只选择一个模板

| 问题类型 | 起点 |
|---|---|
| 通用 2–4 方案 | `assets/templates/choice-grid.html` |
| 页面或导航 | `assets/templates/ui-compare.html` |
| 系统边界和数据流 | `assets/templates/architecture-compare.html` |
| 状态与审批路径 | `assets/templates/process-compare.html` |

正常情况下不要同时读取多个模板，也不要先读全部成品案例。

生成 HTML 片段时确保：

- 所有选项都含 `data-choice` 与 `data-label`；
- 不依赖互联网资源；
- 不把示意数字写成实测；
- 视觉尺度一致；
- 360px 宽度仍可阅读；
- 不写不必要的自定义 JavaScript；
- 文件尽量控制在 120 KB 内。

## 阶段 6：一条命令展示

先把片段写到可写的临时文件，然后运行：

```bash
python3 -I -S "$SKILL_DIR/scripts/companion.py" show \
  --project-dir "$PROJECT_ROOT" \
  --source "/absolute/path/to/hero-direction.html" \
  --name "hero-direction" \
  --open
```

`show` 会：

1. 复用健康会话或启动新服务；
2. 原子发布页面；
3. 在发布完成后请求打开浏览器；
4. 返回 URL、进程和页面元数据。

普通流程不需要再运行 `start`、`status`、`paths` 或单独的 `publish`。这些命令只用于高级控制和故障处理。

## 阶段 7：把控制权交给用户

发布后在聊天中只说三件事：

1. 当前页面只解决什么；
2. 建议观察哪几个差异；
3. 可以点击卡片，也可以直接在聊天中回答。

随后结束当前轮次，不继续追加多个问题。

## 阶段 8：读取并解释事件

下一轮首先运行：

```bash
python3 -I -S "$SKILL_DIR/scripts/companion.py" events \
  --project-dir "$PROJECT_ROOT" --tail 20
```

事件可能包含：

```json
[
  {
    "id": 3,
    "type": "choice",
    "choice": "product-first",
    "label": "产品优先",
    "screen": "...hero-direction.html"
  },
  {
    "id": 4,
    "type": "note",
    "note": "保留这个结构，但缩短首屏文案。",
    "screen": "...hero-direction.html"
  }
]
```

处理顺序：

1. 用户在聊天中的明确修正；
2. 最近一次有效选择；
3. 该选择之后的备注；
4. 更早事件只作为历史。

没有事件时不要卡住，允许用户直接在聊天中选择。

## 阶段 9：逐步收敛

理想迭代：

```text
首屏入口 → 导航结构 → 信息密度 → 整合稿
```

每次迭代都减少未知量。不要重新呈现已经解决的全部方案，也不要把已否决方向换个名字重新引入。

## 阶段 10：固化结论

最终文档至少记录：

- 目标与约束；
- 用户选择及理由；
- 被放弃方案及原因；
- 相关 HTML 页面文件名；
- 实现边界和后续事项；
- 未经验证的数据假设。

优先写入项目现有设计文档；没有合适位置时再创建明确命名的文档。

## 阶段 11：关闭

```bash
python3 -I -S "$SKILL_DIR/scripts/companion.py" stop \
  --project-dir "$PROJECT_ROOT"
```

会话文件默认保留供回看。需要确认时再运行 `status`，不要把 `status` 加进每次正常流程。

## 最小状态清单

```text
[ ] 已理解项目上下文
[ ] 已定义单一视觉决策
[ ] 已处理用户同意
[ ] 只读取一个匹配模板
[ ] 页面只有 2–4 个真实方案
[ ] data-choice / data-label 完整
[ ] 已通过 show 发布并请求打开
[ ] 已读取最近结构化事件
[ ] 选择已写回项目文档
[ ] 本地服务已停止
```
