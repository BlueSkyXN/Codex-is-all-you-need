# 背景、价值与复杂度

## 结论

这个功能并不神秘：它是一个 **Agent 生成视觉页面 → 本地浏览器展示 → 用户点击 → Agent 读取事件** 的小型闭环。

实现复杂度属于中等。真正需要处理的不是“画一张网页”，而是四件事同时成立：

1. 只在视觉确实有价值时启用；
2. 让页面稳定地自动更新；
3. 把浏览器选择可靠地带回对话；
4. 不把生成的页面误称为模型内部思维。

本 Skill 采用可维护的 L2 方案：Python 标准库本地服务、短轮询、sandbox iframe、JSONL 事件和原子发布。

## 1. 功能模型

```text
用户提出设计问题
        │
        ▼
Agent 判断是否值得视觉化
        │
        ▼
生成 HTML / CSS / SVG
        │
        ▼
本地伴侣展示最新页面
        │
        ▼
用户点击方案或提交备注
        │
        ▼
事件追加到 events.jsonl
        │
        ▼
Agent 下一轮读取并继续收敛
```

聊天仍然负责：需求、约束、推理结论、文字说明和最终决策记录。浏览器只负责那些依赖空间、层级、比例、流程或视觉密度的问题。

## 2. 它解决什么问题

核心问题是**表达介质不匹配**。

文字擅长：

- 解释因果；
- 罗列规则；
- 记录约束；
- 对比少量明确参数。

浏览器擅长：

- 同尺度展示多个布局；
- 表达节点、路径、分支和系统边界；
- 呈现信息层级与视觉重心；
- 让用户直接点选一个方案。

因此，价值不在“多一个界面”，而在于减少设计讨论中的语言歧义。

## 3. 它不是什么

它不是：

- 模型思维链查看器；
- Token 流、注意力权重或神经网络状态可视化；
- 浏览器自动化框架；
- 自动生成正确设计的保证；
- Figma、白板或正式原型平台的完整替代；
- 性能、成本和可用性测试工具。

用户看到的是 Agent 主动生成、可审阅、可否决的外部产物。

## 4. 复杂度分层

| 层级 | 能力 | 工程复杂度 | 适用情况 |
|---|---|---:|---|
| L0 | Agent 只写一个 HTML 文件 | 低 | 用户愿意手动打开和刷新 |
| L1 | 本地服务持续展示最新 HTML | 中低 | 需要自动发现新页面 |
| L2 | 浏览器点击写入结构化事件 | 中 | 需要完整的双向决策闭环 |
| L3 | WebSocket、断线恢复、页面历史 | 中高 | 高频实时更新或长会话 |
| L4 | 多用户、远程访问、数据库、身份体系 | 高 | 需要产品化协作平台 |

本 Skill 位于 **L2**。它已经能完成视觉决策闭环，但有意不承担 L3/L4 的实时协作和运维成本。

## 5. 为什么使用短轮询

视觉方案通常每几十秒或几分钟更新一次，而不是每秒持续变化。浏览器前台约 1 秒、切到后台后约 5 秒一次的本地 `/api/latest` 查询具有几个优势：

- 不需要第三方 WebSocket 依赖；
- Windows、macOS 和 Linux 的差异更少；
- 进程和断线状态更简单；
- 便于通过标准 HTTP 工具测试；
- 页面不在前台时自动降低查询频率；
- 对单用户本地页面负担较小。

当需求演变为多人同步白板、连续动画、实时光标或高频流式更新时，再升级 WebSocket 更合理。

## 6. 关键设计原则

### 6.1 按问题启用，而不是按会话启用

一个任务中可能同时存在文字问题和视觉问题。只有在进入具体视觉决策时才打开浏览器，不要把每次头脑风暴都自动变成网页生成。

### 6.2 一屏一决策

如果同一屏同时改变导航、配色、内容、功能和交互，用户无法判断自己究竟选择了什么。每屏只改变一个主变量。

### 6.3 方案必须真实不同

真实差异通常来自：

- 信息优先级；
- 页面或系统边界；
- 交互路径；
- 流程分支；
- 视觉密度；
- 数据流方向。

只换标题、图标或颜色，不构成独立方案。

### 6.4 事件必须结构化

“我喜欢第二个”在多轮对话中很容易失去上下文。事件至少要记录：

- 稳定 `choice`；
- 人类可读 `label`；
- 来源 `screen`；
- 时间与递增 `id`；
- 可选备注。

### 6.5 本地优先

服务默认只绑定 `127.0.0.1`。页面、事件和会话状态保存在项目临时目录，不依赖云端服务。默认连续 2 小时没有页面加载、资源加载或用户事件后自动关闭；纯轮询不延长寿命，避免无人使用的标签页长期保活后台进程。

## 7. 何时值得使用

适合：

- 首页、后台、控制台和编辑器布局；
- 系统架构、RAG 边界、消息流和模型路由；
- 审批流程、状态机、异常路径和实体关系；
- 需要用户直接点选的 2–4 个视觉方向。

不适合：

- “接口字段叫什么”；
- “A 成本 10，B 成本 12”；
- “解释这段代码”；
- “是否支持导出 CSV”；
- 只需要一张普通 Markdown 表格的参数比较。

## 8. Token 与资源成本

本地服务本身消耗很少。主要成本来自 Agent 生成和反复修改 HTML、CSS、SVG 与多套方案。

控制成本的方法：

1. 先用文字确定范围和约束；
2. 每次只生成一个决策页面；
3. 优先复用 `assets/templates/`；
4. 使用片段而不是重复完整 HTML 框架；
5. 不为纯文字问题制作视觉页面；
6. 方向确定后立即停止横向发散。

当前实现还采用两项轻量化措施：

- 正常路径用 `show` 合并启动、发布和打开，减少 Agent 命令往返；
- 后台 Python 使用 `-I -S` 隔离模式，避免用户级 site 包和启动钩子放大常驻进程。

## 9. 什么时候应该继续升级

只有出现以下明确需求时，才考虑超出本 Skill：

- 多人同时操作；
- 需要跨机器或公网访问；
- 需要长期页面历史与检索；
- 需要画布批注、拖拽排序或协同光标；
- 页面更新频率高到轮询明显不足；
- 需要与 Figma、白板或设计系统双向同步。

在这些需求出现之前，L2 往往是成本和价值最平衡的形态。
