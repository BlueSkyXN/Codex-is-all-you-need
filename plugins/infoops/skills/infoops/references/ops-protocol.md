# 操作协议

Agent 执行 InfoOps 操作时的步骤序列。按操作对象分组，每个场景定义"收到请求后做什么"的完整流程。

操作前先读 SKILL.md 确认铁律和 reference 导航。

---

# 一、系统级操作

## 1.1 初始化（最小启动）

```
1. 确认 lark-cli 可用
   command -v lark-cli && lark-cli --version

2. 创建目录
   mkdir -p workspace/infoops

3. 创建 MISSION.md
   → 填入身份、优先级（可以简写，后续完善）

4. 创建 ID-MAP.md
   lark-cli im +chat-list --as user --types p2p,group
   → 按 Internal Groups / External Groups / Internal Chats / External Chats / Bots / Self 分区填入

5. 创建 PULSE.md
   → 只写 Active 和 Directory 两个分区
   → 其余分区写（无）占位

→ 系统可运行，后续按需创建 threads/ sources/ 等
```

### 渐进完善时间表

```
第一周：
  → 创建 threads/ sources/ archive/ 目录结构
  → 判定7天内我有发言的群 → 建群档案
  → 判定30天内我有发言的人 → 建人档案
  → 拉取重点群近7天消息 → 识别活跃线程 → 创建线程文件
  → 首次完整 PULSE 刷新

第二周：
  → 配置 cron 任务
  → 识别需要追踪的云空间文件 → 建 _index 或追踪文件
  → 跑通每日自动刷新流程
  → 部署 _RULES.md

第三周+：
  → 根据实际数据校准信号阈值
  → 按需升格/降格对象
  → 积累足够数据后，跨源关联开始有价值
```

---

## 1.2 PULSE 全量刷新

触发：每日 cron、用户问"帮我看看最近什么情况"且 PULSE 已过期。

```
1. 读 MISSION.md 获取数据源清单和优先级
2. 读 signal-model.md 获取信号节奏判定规则

3. 采集 IM（按优先级）
   a. 读 lark-cli skills read lark-im
   b. 判定「7天内我有发言的群」
      → lark-cli im +messages-search --sender <我的open_id> --start <7天前> --as user
   c. 对高优先级群逐个 +chat-messages-list
   d. 对每个群：排除过滤 → 信号节奏判定 → 更新 DIGEST（如有独立档案）

4. 采集云空间（按优先级）
   a. 读 lark-cli skills read lark-drive
   b. lark-cli drive +search --edited-since 1d --as user
   c. 对命中的文件：检查是否在追踪范围 → 更新变更日志 → 信号判定

5. 检查线程待办
   a. 扫描 threads/active/ 下所有文件
   b. 标记逾期待办
   c. 检查是否有 Cooling 线程（14+ 天无动静）

6. 跨源信号关联
   a. 同一线程涉及的群/人/文档，任一 Surge → 整条线程提升
   b. 同一人 24h 内跨群讨论同一话题 → 标记跨群热点
   c. 文档变更 + 群讨论同时发生 → 检查关联

7. 合成 PULSE.md
   a. 按 🔥→⚡→🌊→🌱→🧊→Directory→Closed 顺序写入
   b. 空分区保留标题，写（无）占位
   c. 更新刷新时间戳
   d. Surge 和 Active 必须有行动建议

8. 如无法访问某数据源
   → 增量模式：只更新能采集到的部分，其余保持原样
   → 在 PULSE header 标注"部分刷新"和未覆盖的数据源
```

---

## 1.3 每日 cron 流程

```
08:00  重点群每日摘要
       → 判定7天内我有发言的群
       → 对每个群 +chat-messages-list
       → 排除过滤 → 更新 DIGEST → 标记信号

08:30  线程待办检查
       → 扫描 threads/active/ 所有文件
       → 标记逾期待办
       → 检查 Cooling 线程

09:00  云空间变更检测
       → drive +search --edited-since 1d
       → 检查命中文件是否在追踪范围
       → 更新变更日志

09:30  PULSE 全量刷新
       → 执行流程 1.2（合成所有数据源信号）

每周一 10:00  PULSE 快照
       → 备份当前 PULSE.md 到 archive/snapshots/

每周五 16:00  Cooling 检查
       → 扫描 14+ 天无动静的对象
       → 建议关闭或干预

每月 1 日 10:00  跨群人物刷新
       → 重新拉取重点群成员
       → open_id 交叉匹配
       → 更新 ROSTER.md

每季度首日  DIGEST 季度归档
       → 归档旧 DIGEST 到 archive/
```

---

## 1.4 ID-MAP 维护

### 新增条目

```
1. 获取 chat_id
   - 群：lark-cli im chats get <chat_id> --as user
   - 单聊：lark-cli im +chat-list --as user --types p2p
   - 自己的 open_id：lark-cli contact +search-user --query <自己的名字> --as user

2. 判断分区
   - 群：lark-cli im chats get <chat_id> → chat_type + tenant_key 判断内/外部
   - 单聊：对方 tenant_key 判断内/外部

3. 追加到对应分区末尾
   - 不排序，不加备注

4. 如果有对应的独立档案 → 检查档案中的 chat_id 是否一致
```

### 定期更新

```
1. 更新 7d消息数 / 30d消息数
   → lark-cli im +messages-search 统计
   → 只更新数值列，不动其他内容

2. 清理不再需要的条目
   → 群已解散 / 人已离职且无存量线程 → 删除该行
   → 不在 _RULES.md 排除列表中的才能删
```

---

## 1.5 写入检查

每次写入 InfoOps 文件前，对照以下清单：

1. [ ] 写的是判断和结论，不是原始数据搬运？
2. [ ] 事件细节在线程里，画像在对象档案里，没有混放？
3. [ ] ID-MAP 没有被添加主观内容？
4. [ ] 标签名称与 ID-MAP 一致？
5. [ ] 涉及外部群 `[Ge:...]` 的内容已检查信息边界？
6. [ ] 待办有具体动作 + 责任人，不是空话？
7. [ ] 时间线倒序？
8. [ ] PULSE 的 Surge/Active 有行动建议？
9. [ ] 云空间文件的变更日志具体到位置？
10. [ ] 被排除的信息（书籍/论文/工具包）没有建追踪文件？

# 二、线程操作

## 2.1 新建线程

触发：发现新事务（从采集中识别、用户指定）。

```
1. 确认线程标题（英文小写短横线命名：<线程名>.md）
2. 读 templates.md 获取线程模板
3. 创建 threads/active/<线程名>.md
   - 填入：涉及对象（标签）、待办、首条时间线
4. 更新 PULSE.md
   - Active Threads 表加一行
5. 关联对象处理
   - 涉及的群/人/文档如不在 PULSE Directory → 加入 Directory
   - 如果对象已有独立档案 → 在档案的线程索引中加一行
6. 写入前对照 1.5 写入检查
```

---

## 2.2 关闭线程

触发：所有待办完成、用户确认关闭。

```
1. 修改线程文件
   - 状态改为 ✅ 已关闭
   - 保留：结论 + 关键决策 + 涉及
   - 删除：时间线和待办
2. 移动文件
   - mv threads/active/<线程名>.md threads/closed/<线程名>.md
   - 如果是升格目录 → 整个目录移到 closed/
3. 更新 PULSE.md
   - 从 Active Threads 移到 Closed Threads
4. 关联对象处理
   - 更新相关档案的线程索引（状态标记改为 ✅）
   - 检查对象的活跃线程数 → 如果降为 0 且满足降格条件 → 触发降格（见 3.5 / 4.4）
```

---

## 2.3 更新线程

触发：采集到与已有线程相关的新事件。

```
1. 定位线程文件
   - 先查 threads/active/
   - 如在 closed/ → 讨论是否重新打开

2. 追加时间线
   - 在对应日期 H3 下追加一行
   - 如当天没有 H3 → 新建（插在最前面，倒序）
   - 只记有意义的事件，不记流水账

3. 更新待办（如有变化）
   - 新增待办 → 追加
   - 完成待办 → 标 [x] + 删除线 + 日期
   - 过期待办 → 标注逾期

4. 更新涉及列表（如有新对象）
   - 新增对象标签
   - 新对象如有独立档案 → 在档案线程索引中加一行

5. 检查是否触发信号变化
   - 新事件是否让线程从 Active → Surge？
   - 是否需要更新 PULSE？
```

# 三、群操作

## 3.1 发现新群

触发：ID-MAP 中出现新 chat_id、用户提到某个群、采集时发现新群。

```
1. 获取群信息
   lark-cli im chats get <chat_id> --as user
   → 得到群名、群主、成员数、chat_type、tenant_key

2. 判断内/外部
   - chat_type + tenant_key 判断
   - 外部群 → 立即标记，后续所有操作附加安全约束

3. 判断是否在采集窗口
   lark-cli im +messages-search --sender <我的open_id> --chat <chat_id> --start <7天前> --as user
   - 7天内我有发言 → 高优先级
   - 7天内我没发言 → 低优先级，只登记 ID-MAP

4. 分流处理
   ┌─ 高优先级 ──→ 登记 ID-MAP + 建群档案（简单版）+ 加入 PULSE Directory
   │
   ├─ 低优先级 ──→ 只登记 ID-MAP
   │
   └─ 排除范围 ──→ 不处理（水群、通知群、bot 群等）

5. 如果是外部群 → 额外执行 3.6 外部群处理
```

---

## 3.2 建群档案

触发：群被判定为高优先级（3.1 步骤 4）。

```
1. 读 templates.md 获取群档案模板（简单版）
2. 创建文件
   - 内部群：sources/groups/internal/<群名>.md
   - 外部群：sources/groups/external/<群名>.md
3. 填入基本信息
   - 定位：一句话
   - 关键成员：从群成员列表中筛选（群主 + 活跃发言者）
   - 沟通规则：外部群必须包含安全约束
4. 登记到 PULSE Directory（Profiled 表）
5. 首次拉取消息 → 写 DIGEST 初版（如已升格）或在档案备注中记录近况
```

---

## 3.3 DIGEST 单次更新

触发：每日 cron、用户问某群近况、手动触发。

```
1. 确认群在采集窗口（7天内我有发言）
   - 不在窗口 → 跳过，或提醒用户该群已不在重点关注范围

2. 拉取消息
   lark-cli im +chat-messages-list --chat <chat_id> --start <上次更新时间> --as user
   → 读 lark-cli skills read lark-im 了解参数细节

3. 排除过滤
   - 系统消息、bot 消息、纯表情回复 → 跳过
   - 与 MISSION 无关的闲聊 → 跳过

4. 提炼判断（不是搬运聊天记录）
   ✗ <人名> 14:32 说："我觉得这个兼容性有问题"
   ✓ <人名>提出 <版本号> 与旧版本存在兼容性风险

5. 写入 DIGEST.md
   - 更新覆盖时间范围
   - 关键讨论：按日期 H3 分组
   - 趋势：活跃度/热点/新活跃/沉默
   - 待关注：需要跟进的事项

6. 信号判定
   - 消息量异常 → Surge？
   - 新话题出现 → Emerging？
   - 如信号变化 → 更新 PULSE 对应分区
```

---

## 3.4 群升格

触发：群关联 ≥2 活跃线程、或群内信息密度高到简单档案装不下。

```
1. 确认满足升格条件（见 taxonomy.md）

2. 创建目录
   mkdir sources/groups/<internal|external>/<群名>/

3. 拆分文件
   - 原 <群名>.md → PROFILE.md（去掉线程索引）
   - 新建 DIGEST.md（拉取近 7 天消息，写首版）
   - 新建 ROSTER.md（从关键成员表展开为三表结构）

4. 删除原 <群名>.md

5. 更新引用
   - PULSE Directory 中路径更新
   - 相关线程中的标签加 ★
   - 相关人档案的跨群足迹路径更新
```

---

## 3.5 群降格

触发：所有关联线程关闭且 90 天无新线程、群已不在7天窗口内。

```
1. 确认满足降格条件
   - 无活跃线程
   - 90+ 天无新线程
   - 不在7天发言窗口

2. 合并文件
   - PROFILE.md 内容精简为简单版 <群名>.md
   - DIGEST.md 最后一版归档到 archive/digests/
   - ROSTER.md 核心成员表合并到简单版的关键成员

3. 删除目录

4. 更新引用
   - PULSE Directory 中更新路径或移到 Tracked 表
   - 相关标签去掉 ★

5. 进一步降格：如简单档案也不再需要
   → 从 sources/ 删除
   → PULSE Directory 中移除或只保留 Tracked 一行
   → ID-MAP 条目保留（ID-MAP 不因降格删条目）
```

---

## 3.6 外部群特殊处理

触发：发现群是外部群（3.1 步骤 2），或已有群被重新判定为外部群。

```
1. 标记
   - ID-MAP 中放入 External Groups 分区
   - 标签使用 [Ge:...] 而非 [Gi:...]

2. 建档时的额外要求
   - 沟通规则必须包含：
     · 禁止暴露内部排期和未发布功能
     · 其他外部群特有的安全约束
   - DIGEST 中涉及内部信息的讨论标注 [内部敏感]

3. 信息边界检查
   - 每次更新 DIGEST 时检查：
     · 我方发言是否有内部信息泄露 → 标注 ⚠️
     · 外部人员是否询问了敏感信息 → 记录在 DIGEST 待关注
   - 涉及的人档案加"识别备注"分区

4. 线程关联
   - 涉及外部群的线程，涉及列表中写 [Ge:...] → Agent 自动应用外部沟通规则
```

# 四、人物操作

## 4.1 发现新人

触发：群消息中出现新发言者、单聊中出现新对象、群成员列表中识别到新的活跃人物。

```
1. 获取身份信息
   - 从消息中提取 open_id
   - lark-cli contact +search-user --query <姓名或open_id> --as user
   → 得到姓名、部门、职位

2. 跨群匹配
   - 用 open_id 在 ID-MAP 和已有人档案中搜索
   - 同一 open_id 出现在多个群 → 合并为同一人
   - 不要凭名字匹配，必须用 open_id

3. 分流处理
   ┌─ 30天内我有发言的单聊对象 ──→ 高优先级 → 建人档案
   │
   ├─ 重点群中的活跃发言者 ──→ 中优先级 → 登记 ID-MAP + ROSTER
   │
   ├─ 重点群中偶尔发言的成员 ──→ 低优先级 → 只登记 ROSTER
   │
   └─ 其他 ──→ 不处理

4. 判断内/外部
   - tenant_key 一致 → 内部
   - tenant_key 不一致 → 外部 → 加"识别备注"分区
```

---

## 4.2 建人档案

触发：人被判定为高优先级（4.1 步骤 3）。

```
1. 读 templates.md 获取人档案模板
2. 创建 sources/people/<人名拼音>.md
3. 填入基本信息
   - 角色关系：与我的工作关系
   - 跨群足迹：在哪些群出现、什么角色
   - 线程索引：关联的线程（如有）
4. 登记到 PULSE Directory（Profiled 表）
5. 如果是外部人物 → 加"识别备注"分区
```

---

## 4.3 人物升格

触发：人关联 ≥3 活跃线程、或跨群出现在 ≥3 个群且行为模式复杂。

```
1. 确认满足升格条件

2. 创建目录
   mkdir sources/people/<人名拼音>/

3. 迁移文件
   - 原 <人名拼音>.md → PROFILE.md
   - 创建 archive/ 子目录

4. 删除原 .md 文件

5. 更新引用
   - PULSE Directory 路径更新
   - 相关线程标签加 ★
   - 相关群 ROSTER 中标注有独立档案
```

---

## 4.4 人物降格

触发：所有关联线程关闭且 90 天无新线程、不再是 30 天单聊窗口内的对象。

```
1. 确认满足降格条件

2. 从目录降为单文件
   - PROFILE.md 精简为 <人名拼音>.md
   - archive/ 内容归档到统一 archive/

3. 删除目录

4. 进一步降格：如单文件也不再需要
   → 从 sources/ 删除
   → PULSE Directory 中移除或只保留 Tracked 一行
   → 群 ROSTER 中保留名字但去掉独立档案标记

5. 更新引用
   - 标签去掉 ★
   - PULSE Directory 路径更新
```

---

## 4.5 跨群人物合并

触发：月度跨群人物刷新时、或日常采集中发现同一 open_id 出现在多处。

```
1. 识别
   - 同一 open_id 在多个群的 ROSTER 中出现
   - 或 ID-MAP Internal/External Chats 中的人同时在某群发言

2. 合并策略
   - 如已有人档案 → 更新跨群足迹表，加新群的行为信息
   - 如无人档案但跨群 ≥3 → 考虑建档
   - 只登记在 ROSTER 中 → 在多个 ROSTER 的跨群出现列标注

3. 注意
   - 不凭名字合并，必须用 open_id
   - 内部人在外部群出现 → 特殊标注
```

# 五、云空间操作

## 5.1 登记新文件（→ L1）

```
1. 确认文件类型（Doc/Sheet/Bitable/PDF）
2. 获取 token
   lark-cli drive +inspect <url_or_token> --as user
3. 在对应 _index.md 中追加一行
   文件名 | 类型 | token | 一句话定位
4. 不建变更日志，不建关注点
```

---

## 5.2 升格为聚合追踪（L1→L2）

```
1. 确认有 ≥2 个相关文件被关联到同一主题
2. 读 templates.md 获取聚合追踪模板
3. 创建 sources/space/<类型>/<主题>.md
   - 列出主线文件表
   - 填入关注点和首条变更日志
4. 从 _index.md 中移除对应文件行
5. 在关联线程的涉及列表中添加标签
```

---

## 5.3 升格为独立追踪（L1→L3 或 L2→L3）

```
1. 确认文件满足条件
   - ≥2 线程关联 / 高频协作 / 用户手动标记
2. 读 templates.md 获取独立追踪模板
3. 创建 sources/space/<类型>/<文件名>.md
   - 填入定位、关注点、协作者、首条变更日志
4. 从 _index.md 或聚合 md 中移除对应条目
5. 如从聚合 md 拆出 → 检查聚合 md 剩余文件是否仍需聚合追踪
6. 登记到 PULSE Directory（Profiled 表）
```

---

## 5.4 云空间降格

### L3→L2（独立追踪 → 聚合追踪）

```
1. 确认降格条件
   - 关联线程全部关闭
   - 30+ 天无变更
   - 存在同主题的聚合追踪 md 可以接收

2. 将文件信息合并到聚合追踪 md
   - 主线文件表加一行
   - 关注点合并
   - 最后几条变更日志合并

3. 删除独立追踪 md
4. PULSE Directory 中移除该文件
```

### L2→L1（聚合追踪 → _index 索引）

```
1. 确认降格条件
   - 聚合追踪中所有文件 60+ 天无变更
   - 无关联活跃线程

2. 将每个文件的基本信息（名字、类型、token、定位）写回 _index.md

3. 删除聚合追踪 md
```

### L1→L0（从 _index 中移除）

```
1. 确认移除条件
   - 文件已被删除 / 已无权限访问 / 确认不再需要关注

2. 从 _index.md 中删除该行

3. 不需要其他更新（L1 没有关联线程或 PULSE 条目）
```

## 5.5 云空间变更日志更新

触发：每日 cron 的云空间变更检测、用户询问某文件近况。

```
1. 检查文件在哪个层级
   - L1（_index）→ 不记变更日志，但检查是否需要升格
   - L2（聚合）→ 更新聚合 md 的变更日志
   - L3（独立）→ 更新独立 md 的变更日志

2. 拉取变更信息
   lark-cli drive +inspect <token> --as user
   → 得到最后编辑时间、编辑者

3. 写入变更日志
   - 格式：<MM-DD>: <谁>做了<什么变更>（<具体位置>）
   - 变更要具体到位置，不写"文档有更新"
   - 不记录格式调整、错别字修正

4. 信号判定
   - 检查 signal-model.md 中的云空间判定规则
   - 如信号变化 → 更新 PULSE
```

# 六、用户指令响应

## 6.1 "帮我看看最近什么情况"

```

2. 检查刷新时间 → 超过 24h 则先触发 PULSE 刷新（1.2）
3. 按 Surge → Active → Emerging 顺序汇报

```

## 6.2 "帮我看看 <某对象> 的情况"

用户问"<群名>最近在聊什么"、"<人名>最近有找我吗"、"<文件名>有什么变化"等。

```
1. 在 sources/ 或 ID-MAP 中定位对象

2. 分情况处理
   ┌─ 有独立档案 ──→ 读档案 + 关联线程 → 汇报
   │
   ├─ 只在 ID-MAP ──→ 临时拉取最新数据 → 汇报 → 问是否需要建档
   │
   └─ 找不到 ──→ 尝试搜索（lark-cli contact / im / drive）→ 问是否需要纳入追踪

3. 如需最新数据 → 读对应 lark-cli 内嵌 skill → 采集

4. 处理采集结果
   - 排除过滤 → 信号判定 → 更新档案（如有）

5. 汇报 + 行动建议
```

---

## 6.3 "帮我盯着 X"

用户要求关注某个群/人/文件。

```
1. 识别对象
   - 在 ID-MAP / sources/ 中查找
   - 找不到 → 先通过 lark-cli 获取信息

2. 判断当前追踪状态
   ┌─ 已有独立档案 ──→ 确认已在 PULSE Active，如不在则加入
   │
   ├─ 只在 ID-MAP ──→ 建档（群/人/云空间对应的建档流程）
   │
   └─ 完全不在系统中 ──→ 登记 ID-MAP + 建档

3. 如用户指定了关注点
   - 建线程跟踪具体事项
   - 或在档案关注点中添加检测规则

4. 调整 PULSE
   - 确保对象出现在 Active 或 Ongoing（取决于性质）
   - 添加行动建议
```

---

## 6.4 "X 不用管了"

用户取消对某个群/人/文件/线程的关注。

```
1. 识别对象和当前追踪状态

2. 分情况处理

   线程：
   → 执行关闭线程流程（2.2）
   → 如用户说"不是关闭，是直接删" → 移到 archive/ 但不写结论

   群/人：
   → 如有独立档案 → 触发降格（3.5 / 4.4）
   → 如只有简单档案 → 直接删除 + 从 PULSE Directory 移除
   → ID-MAP 条目保留（除非用户明确说删）

   云空间文件：
   → 触发降格到 L0（5.4 L1→L0）
   → 如在聚合/独立追踪中 → 逐级降格

3. 更新 PULSE
   - 从 Active/Ongoing 中移除
   - 如有必要标注"用户主动取消关注"

4. 不追问"你确定吗" → 直接执行，用户说了就做
```

---

## 6.5 "这两个是同一件事"

用户指出两个线程/对象其实是同一个。

```
1. 确认要合并的对象

2. 线程合并
   - 选择保留哪个（内容更丰富的）
   - 将另一个的时间线、涉及、待办合并到保留的
   - 去重（同一事件不重复记录）
   - 被合并的线程关闭，结论写"已合并到 <保留的线程>"

3. 对象合并（如两个群档案实为同一群）
   - 保留信息更完整的
   - 更新 ID-MAP（如有重复 chat_id → 去重）
   - 更新所有引用（线程标签、PULSE 路径）
```

## 6.6 "帮我总结一下 X 这段时间的情况"

用户要求回顾性总结。

```
1. 确定时间范围
   - 用户指定 → 按指定范围
   - 未指定 → 默认近 7 天

2. 收集信息
   - 读对象档案（DIGEST / 变更日志 / 线程时间线）
   - 如果本地信息不够 → 从 lark-cli 补充采集

3. 生成总结
   - 按时间线展示关键事件
   - 标注信号变化（从 Active 变成 Cooling 等）
   - 给出行动建议

4. 不写入文件，直接口头汇报
   - 除非用户说"写下来" → 写入对应的 DIGEST 或新建总结文档
```
