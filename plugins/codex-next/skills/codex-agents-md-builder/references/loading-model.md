# Codex AGENTS.md 加载模型详解

SKILL.md 的核心世界观四行讲清了"是什么"。本文件讲"为什么"和"边界条件"。

## 1. 启动时的项目指令发现

Codex 启动时，会读取 project root 到当前工作目录路径链上的 AGENTS.md；越靠近当前工作目录的文件，适用范围越近，冲突时优先级越高。

但绝大多数团队 workflow 是：**Codex 从仓库根目录启动**。

这导致的实际效果：

- 根目录 AGENTS.md **自动进入启动上下文**
- 子目录 AGENTS.md **不在自动加载路径上**——它们不会默认进入上下文
- 全局 / 用户级 / 工作区级的其他指令文件会按 Codex 的层级规则参与，与本 skill 设计无关

子目录 AGENTS.md 不在根启动期出现，并不意味着它们没用。它们仍然会在 Codex 从该子目录或其后代启动时自动参与；从根启动的团队 workflow 中，能否被及时消费主要取决于根 AGENTS.md 是否给出清楚的按需读取指引。

## 2. 子目录 AGENTS.md 何时才会被消费

常见路径有三种：

1. Codex 从该子目录或其后代目录启动（少见）
2. 根 AGENTS.md 明确指引 agent："修改 `services/payments/` 前，先 cat `services/payments/AGENTS.md`"
3. 用户在会话中显式要求读取该文件

所以根 AGENTS.md 必须扮演 **router** 角色：

- 提供完整目录地图
- 标注每个子目录是否有本地 AGENTS.md
- 写明"什么时候应该 cat 它"

否则从根启动时，子目录卡片很容易被漏读，存在但没有发挥实际约束效果。

## 3. 字节预算

Codex 用 `project_doc_max_bytes` 控制项目指令读取预算，官方示例默认值是 **32768 字节（32 KiB）**。

注意：

- 这个预算**只对自动加载的项目指令生效**——也就是从 cwd 向上找到的 AGENTS.md
- 子目录卡片如果是按需 cat，**不进入这个预算**
- 但 cat 进上下文后，仍占用对话 token

实际取值建议（按 UTF-8 文件字节估算，**不**按 token / 段落 / 行数）：

| 文件 | 目标 | 硬上限 |
|---|---|---|
| 根 AGENTS.md | 8–16 KiB | 25 KiB |
| 子目录 Domain card | 0.5–2 KiB | 4 KiB |
| 子目录 Guardrail card | 0.5–2 KiB | 4 KiB |
| 生成 / 只读目录卡片 | < 0.5 KiB | 1 KiB |

超过硬上限要么拆 references 子文件（仓库内可以用 Markdown 链接），要么裁掉重复 / 空泛内容。

## 4. 沙箱与命令约束

Codex 默认沙箱特征：

- 无网络
- 非交互（不能等用户输入）
- 非 sudo
- 仅访问 workspace（不能修改用户级文件、外部服务）

**Validation 命令落入 AGENTS.md 时必须区分两类**：

| 类别 | 处理方式 |
|---|---|
| 可在默认沙箱运行（单元测试、lint、typecheck、纯文件操作） | 直接列出，不加额外标注 |
| 需要网络 / Docker / 数据库 / 外部服务 / 凭据 / sudo / 交互 | **显式标注限制**，并明确"在受限环境下跳过或改用 X" |

**不要把需要外部依赖的命令当作默认验证步骤**。如果 e2e 测试需要数据库，那它就不是"完成验证"的默认动作，而是"需要数据库可用时再跑"的可选验证。

## 5. AGENTS.override.md

override.md 用于**屏蔽**同目录的 AGENTS.md（不是"补充"，是"替换"）。

如果目标目录已经存在 `AGENTS.override.md`，继续创建或更新同目录普通 `AGENTS.md` 通常没有效果。此时必须暂停并问用户是维护 override，还是移除 / 改造现有 override 策略。

默认**不创建**。只有以下情况才考虑：

- 仓库已有 AGENTS.md 由其他系统/团队维护，本 skill 不能直接改它
- 需要在临时分支/特定场景下覆盖整套规则
- 用户明确要求

其他情况一律继续用 AGENTS.md 增量维护。

如果本 skill 判断需要创建或修改 override.md，**必须停下来询问用户**，不要自动创建或改写。

## 6. 与 README / CONTRIBUTING 的关系

AGENTS.md 是**给 agent 看的项目指令**，不是用户文档。两者面向不同读者：

| 维度 | AGENTS.md | README |
|---|---|---|
| 读者 | Codex agent | 人类开发者 |
| 触发 | 启动时自动加载 | 人主动打开 |
| 重点 | 命令、约束、目录地图、按需 cat 协议 | 项目介绍、安装、用法、贡献 |

**关键命令必须复述进 AGENTS.md**，不能只写 "see README"。原因：

- Codex 不会主动跳读 README 来推断命令
- 依赖 README 等于丢失启动上下文
- 写 "see README" 等于强制 Codex 额外开一次工具调用，且读到的内容可能与 README 当前状态不匹配

可以的写法：

- AGENTS.md 列出命令和最小说明
- 详细解释、背景、设计动机放 README

## 7. 冲突优先级

按顺序处理：

1. **根 AGENTS.md 与子目录 AGENTS.md 冲突** → 子目录优先（更接近适用范围）
2. **新 Codex 加载模型与已有 AGENTS.md 内容冲突** → 新模型优先
3. **删除 / 压缩 / 改写已有人工维护内容** → 必须在最终汇报中列摘要并说明理由
4. **冲突无法安全判断** → 暂停并请求用户确认，不要静默删除

不要假装看不到冲突。看似人工维护的重要规则被删时，**显式说出来**比偷偷删更安全——用户可以选择恢复。
