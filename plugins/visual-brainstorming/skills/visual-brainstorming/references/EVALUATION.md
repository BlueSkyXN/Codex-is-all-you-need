# 技能评估

## 1. 触发评估

### 应触发

1. “在浏览器里并排展示三套仪表盘布局，我要点选。”
2. “把两种 Agent 架构的数据流画出来比较。”
3. “用可交互流程图比较串行审批和风险分级审批。”
4. “我看文字理解不了，做几个本地 HTML 模型路由示意图。”
5. “给产品首页做三个 mockup，在本地网页展示。”

### 不应触发

1. “把三个模型的价格做成 Markdown 表格。”
2. “解释这段 Python 为什么报错。”
3. “这个 PR 是否应该合并？”
4. “展示你的隐藏思维链。”
5. “给我写一份会议纪要。”

### 边界提示

“比较三个模型”不足以触发。需要继续判断：

- 文字参数 → 聊天表格；
- 路由、架构、数据流 → 浏览器；
- 真实指标图表 → 先取得数据；
- 模型选择器 UI → 浏览器 mockup。

## 2. 流程评分

每项 0–2 分，总分 24 分。

| 项目 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| 启用时机 | 无关时启动 | 稍早或稍晚 | 首个真实视觉问题时启动 |
| 用户同意 | 未处理 | 说明不完整 | 明确请求视为同意；隐式触发只询问一次 |
| 上下文理解 | 未读取 | 只读部分 | 目标、受众和约束明确 |
| 单屏聚焦 | 多变量混杂 | 有次要干扰 | 一个主决策 |
| 方案差异 | 表面不同 | 部分不同 | 结构上真实不同 |
| 控制变量 | 无控制 | 部分一致 | 非决策变量保持一致 |
| 数据诚实 | 编造 | 标注不完整 | 事实、估算、示意清楚 |
| 可选择性 | 无事件 | 部分可点 | 稳定 ID 与标签完整 |
| 可访问性 | 难读 | 基本可用 | 键盘、移动端、对比度良好 |
| 事件处理 | 靠猜 | 读取但不处理修正 | 结构化读取且尊重用户修正 |
| 收敛 | 重复发散 | 部分收敛 | 每屏减少未知量 |
| 收尾 | 服务和结论遗留 | 只完成一项 | 固化结论并停止服务 |

建议合格线：20/24，且以下项目不得为 0：

- 用户同意；
- 数据诚实；
- 事件处理；
- 收尾。

## 3. 结构检查

Skill 目录不携带开发期测试目录或校验脚本；这些内容由宿主 Plugin 仓库维护。发布 Plugin 时至少确认：

- Plugin 中只有一个 `skills/visual-brainstorming/` Skill 目录；
- Skill 目录内部不包含第二个 `.codex-plugin/`、`plugin.json` 或 marketplace 文件；
- `SKILL.md` 的 `name` 与目录名一致；
- `scripts/companion.py`、浏览器外壳、注入助手和四类模板存在；
- README、NOTICE 与 SOURCES 都明确写出参考对象及独立实现关系；
- 文档没有指向缺失文件；
- 示例和模板不依赖 CDN、远程字体或在线图片。

自动化回归放在宿主仓库的 `tests/` 中，不随 Skill 目录分发；公开发布前必须实际运行，而不是只保留人工清单。

## 4. 人工浏览器检查

### 桌面

- 1440×900；
- 三列页面不拥挤；
- 选择状态明显；
- Shell 能显示当前页面名和连接状态；
- 点击后出现“已记录”提示。

### 移动

- 390×844；
- 卡片改为单列；
- 无水平滚动；
- 标题不过度截断；
- 点击区域足够大。

### 暗色模式

- 文字和边界可读；
- SVG 使用 `currentColor` 或适配变量；
- 选中状态仍清楚；
- 不出现纯白刺眼区域。

## 5. 运行验收

手工执行：

```bash
python3 -I -S scripts/companion.py demo --project-dir /tmp/vb-demo --open
```

依次确认：

1. 无 key 的根页面只返回不含会话信息的 bootstrap，受保护 API 返回 403；
2. 带 `#key=...` 的 URL 可打开，刷新当前 tab 后 `sessionStorage` 可恢复；
3. 内置 demo 自动展示；
4. 点击方案后写入 `events.jsonl`；
5. `events --tail 20` 同时返回最近选择及后续备注；
6. 发布新页面后浏览器自动更新；
7. `stop` 后 `status` 为 `running: false`；
8. `../` 路径穿越、点文件、符号链接和 HTML 文档不能通过 files 路由读取；正确会话页面可用 `../files/<path>` 加载辅助资源；
9. `/api/session` 不包含 `control_url` 或会话 key；
10. 设置较短 `--idle-timeout` 后，即使持续请求 `/api/latest`，没有页面加载或用户事件时服务仍能自动停止；
11. 后台启动命令包含 Python `-I -S`，不会继承用户级 site 包。
12. `prune` 默认 dry-run，保留当前/活动 session；apply 必须带 dry-run 的精确 plan，候选变化后旧 plan 被拒绝。
13. `/api/latest` 返回非空 per-screen bridge，`/screen/` 注入相同 bridge，切屏后 bridge 改变，外壳拒绝 stale/wrong bridge。
14. `server.log` 不含 session key，routine 轮询/拒绝请求不让日志线性增长，诊断日志不超过 256 KiB。
15. 远程 URL 不能使用公网、hostname 或另一台内网主机的地址；旧版健康服务显示 `compatible: false`，必须 `--new` 才替换；Cookie 即使包含正确 key 也不能通过认证。

## 6. 输出验收

最终交付应包含：

- 已选方向及影响；
- 被放弃方向及理由；
- 仍待确认的问题；
- 结论写回的项目路径；
- 本地服务是否停止；
- 不把页面描述成模型内部思维。

## 7. 领域案例评估

新增案例时，每个案例至少回答：

- 用户要决定什么？
- 哪些变量保持不变？
- 方案为什么真实不同？
- 是否有未经验证的数据？
- 选择如何写回项目？

案例若只是“模板换字”，不应纳入 `examples/`。
